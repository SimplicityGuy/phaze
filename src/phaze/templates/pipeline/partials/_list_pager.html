{# The reusable list pager (phaze-a6hm.9): showing-range · page numbers · page-size selector. Adapted from proposals/partials/pagination.html, whose layout and affordances this reproduces (same three-zone bar, same "Showing X-Y of N", same 7-link window with ellipses, same Per-page choices). The substantive change is how the URLs are built. The legacy pager spelled ?page={{ }}&status={{ }}&q={{ }}&sort={{ }}&order={{ }}&page_size={{ }} out by hand SIX times, once per control. That is six independent chances to forget a parameter, and a pager that "preserves the filter" only in the sense that six separate authors each remembered to. Here every control calls view.query(...) and states ONLY what it changes, so preserving filter, search, page size and sort is structural rather than repeated. It is also why phaze-a6hm.10 can add sorting without touching this file at all: sort/order are already carried. The legacy interpolation was additionally wrong for real inputs -- a search containing a space or an `&` produced a broken or truncated URL, because `q={{ search_query }}` never encoded anything. view.query() goes through urlencode. Context contract: * view -- ListViewState; the current state and the URL builder. * pagination -- Pagination (services/proposal_queries.py): .page .page_size .total .total_pages .has_prev .has_next .start .end. * pager_url -- path to request (e.g. "/s/propose"). * pager_target -- css selector of the swap target (e.g. "#propose-workspace-list"). POST-ACTION TOTALS (phaze-a6hm.9 acceptance): this partial renders INSIDE the list container, so it is re-rendered by the same swap that re-renders the rows and always reports the totals of the view it is sitting in. Rendering it OUTSIDE the container is the specific defect phaze-7j50 fixed on the legacy view, where a bulk approve/reject replaced the rows but left "Showing 1-50 of 312" from before the action. phaze-a6hm.11 adds bulk actions to this workspace and inherits the correct behaviour for free -- provided the pager stays inside the container. with an aria-label, and the current page carries aria-current="page", so the control is navigable and its state announced rather than conveyed by background colour alone (WCAG 1.4.1) -- the legacy markup coloured the active page and said nothing.
