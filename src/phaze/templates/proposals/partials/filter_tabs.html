{# Status filter tabs -- ADAPTED (phaze-a6hm.2), not rewritten. The visual language is unchanged from the legacy proposals view: the same bottom-bordered nav, the same four All/Pending/Approved/Rejected tabs in the same order, the same blue underline on the active tab and the same pill-shaped count badge. What changed is that the markup no longer hardcodes WHERE it points. It used to build its own `/proposals/?status={{ value }}` URL and target `#proposal-content`, which is what tied it to a single page and left it orphaned when the v7 cutover replaced that page. Context contract (all required; no defaults, so a caller that forgets one fails loudly at render rather than silently emitting tabs that navigate somewhere wrong): * view -- a ListViewState (routers/view_state.py). Supplies BOTH the active-tab test (view.status) and the URL for every tab, via view.query(...). Because query() re-emits the whole state, clicking a tab preserves the search text, the page size and the sort -- the caller does not have to remember to thread them, and phaze-a6hm.10's sort params ride along with no edit to this file. * stats -- ProposalStats; the .total/.pending/.approved/.rejected counts in the badges. * tabs_url -- the path tabs navigate to (e.g. "/s/propose"). * tabs_target -- css selector of the swap target (e.g. "#propose-workspace-list"). `page=1` is reset on every tab: if the operator is on page 3 of Pending and clicks Approved, page 3 of the new filter is not where they meant to land, and may not exist at all. hx-push-url="true" puts the filtered URL in the address bar so the view is bookmarkable and survives Back. That is safe here ONLY because tabs_url is a real full-page route: a restore of the pushed URL re-renders the whole document (response_shape.py rule 2). Do NOT repoint this at a fragment-only endpoint. #}
