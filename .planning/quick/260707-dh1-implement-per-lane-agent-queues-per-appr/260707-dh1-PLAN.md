<objective>
Split the nox file-server agent's single shared SAQ worker into 4 per-lane workers
(analyze / fingerprint / meta / io) so I/O offload and cheap analysis stop being
head-of-line-blocked behind CPU-bound essentia backlog. Implement EXACTLY the
approved design (`docs/superpowers/specs/2026-07-07-agent-queue-lanes-design.md`)
— no redesign.

Purpose: fairness / no head-of-line blocking across task types, on-demand I/O
offload "then and there," and honest core accounting (CPU-bound lanes sum to <=
cores; essentia/TF pinned single-threaded).

Output: a `LANE_TASKS` single-source-of-truth map, lane-parametrized worker
settings, lane-aware queue naming, per-lane concurrency knobs, a 4-service homelab
compose split (+ single-lane compute agent), a legacy-queue drain mechanism, docs,
and tests holding the totality/routing/heartbeat contracts at 90% coverage.

MIGRATION/DRAIN MECHANISM (chosen): a transitional legacy-queue consumer via the
back-compat all-mode. Producers flip to lane queues on deploy. The pre-existing
single `worker` service is kept temporarily with PHAZE_AGENT_LANE UNSET (all
functions) on the legacy un-suffixed `phaze-agent-nox` queue and
PHAZE_AGENT_HEARTBEAT=false, so in-flight jobs of ANY type finish IN PLACE (no
re-enqueue → no duplicate-active hazard). New work never lands on the legacy queue,
so it drains monotonically; when `phaze-agent-nox` reports 0 queued + 0 active the
operator removes the drain service. Rejected the one-off re-enqueue option because
re-driving an already-ACTIVE multi-hour process_file onto a lane queue would
duplicate a running job (deterministic-key dedup guards queued enqueues, not an
active job on a different queue name).

COMPUTE/X86 AGENT DECISION: the cloud/x86 compute agent does NOT get a 4-lane
split. It is media-less and analysis-only — its ONLY task is process_file (received
via the /pushed callback in agent_push.py using queue_for(agent_ref)). Because the
producer now targets lane-suffixed queue names uniformly, the compute agent must
consume the analyze lane, so docker-compose.cloud-agent.yml gets a ONE-LINE
addition (PHAZE_AGENT_LANE=analyze). The other 3 lanes (fingerprint/meta/io) would
be permanently empty on a compute host, and the I/O-starvation / head-of-line
problems the design solves are file-server-only. k8s burst pods are untouched (they
are one-shot phaze.job_runner, not the SAQ worker).

HEARTBEAT DEPTH CAVEAT (A1, by design): heartbeat.py reads ctx["worker"].queue —
now only the analyze lane's depth in the analyze-lane worker. The heartbeat's
queue_depth field is therefore analyze-lane-only, NOT the whole agent. This is
intentional and acceptable: the authoritative all-lane in-flight figure is the
dashboard's get_queue_activity (Task 3), which sums every lane. Documented in
docs/agent-queue-lanes.md; the heartbeat is not changed to sum lanes (it needs only
liveness, and cross-lane queue reads would add coupling for a cosmetic field).
</objective>
