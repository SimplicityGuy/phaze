## Gaps (new — from integration audit)

**GAP-01 (WARNING — deferred, == v2 PROV-01).** `tasks/reenqueue.py:374` `recover_orphaned_work` still routes held `AWAITING_CLOUD` orphan re-drives through `select_active_agent(session, kind="compute")` (Phase-49 code, not visited by phases 72–76). In an N-compute registry this bypasses rank/cap tiering (MCOMP-04) and per-entry `agent_ref` binding (MCOMP-02) and stamps no `cloud_job.backend_id`, so recovery-routed work is not attributed/counted in the lane snapshot. **Severity WARNING, not blocker:** it is fail-safe (catches `NoActiveAgentError`, never raises), off the main dispatch path (which is fully per-agent and verified), and only reachable via the crash/queue-loss orphan-recovery path with a compute agent online. Untested even in the ≤1-compute world. **Disposition: already scoped out as v2 PROV-01** ("N-compute per-agent orphan recovery — a feature with Phase-45-class over-enqueue risk, not a fix"; STATE.md roadmap-evolution note, 2026-07-06). No new phase required for 2026.7.2 close; carry into the v2 PROV backlog.
