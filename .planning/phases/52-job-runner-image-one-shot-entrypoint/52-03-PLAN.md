<objective>
Package the entrypoint into the publishable x86 Job-runner image and wire it into the release
workflow. Two deliverables: (1) `Dockerfile.job` — `FROM` the freshly-built x86 api image (which
already carries Python 3.14 + essentia + native libs + the `phaze` package + uv), with ZERO new
pip dependencies, baking the operator's internal CA and setting `CMD ["uv","run","python","-m","phaze.job_runner"]`
(KJOB-01, KJOB-05); (2) a `build-job-runner` job in `.github/workflows/docker-publish.yml` that
is `needs: build-and-push`-gated — NOT a sibling matrix row — because `Dockerfile.job` builds
`FROM` the just-pushed api tag, which a parallel matrix entry cannot depend on (D-04, Pitfall 1).
Clone the existing `parity-golden-x86` precedent.

Static deployment guards prove the contract without a live build: no `pip install`/`uv add` line
in `Dockerfile.job`, the workflow job is needs-gated, and the CMD targets `phaze.job_runner`.

Purpose: make the execution unit pullable from GHCR in lockstep with the rest of phaze.
Output: `Dockerfile.job`, the workflow job, and `tests/test_deployment/test_job_image.py`.

Note: this plan has NO source-code dependency on Plan 02 — the Dockerfile is a text CMD string,
the workflow is YAML, and the guards are static grep/parse. It runs in Wave 1 in parallel with
Plan 01. The image only becomes runnable once Plan 02's module ships, but nothing here imports it.
</objective>
