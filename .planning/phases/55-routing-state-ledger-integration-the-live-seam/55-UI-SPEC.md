# Phase 55 — UI Design Contract

> Visual and interaction contract for the **KROUTE-06 pipeline-dashboard admission-state cards** — a thin,
> server-rendered read over the new `cloud_job.cloud_phase` column. CONTEXT **D-04 LOCKS** that these cards
> mirror the Phase 54 `inadmissible_card.html` pattern, so this is a **reuse/extension** contract, not a
> net-new visual language. Every decision below references or extends an existing dashboard partial.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope

The **only** UI surface in Phase 55 is the admission-state cards. They display **counts per `cloud_phase`**
(`queued_behind_quota` / `admitted` / `running` / `finished`) for Kueue cloud jobs, kept live by the existing
5s `/pipeline/stats` OOB poll. The cards are **read-only observability** — there is no new CTA, form, or
destructive action in this surface. (The "Backfill to K8s" operator action from D-03/KROUTE-05 reuses the
existing backfill button/endpoint and is contracted in the phase plan, not here.)

The existing `inadmissible_card.html` (Phase 54, amber `role="alert"`) is **NOT re-specified** here — it stays
exactly as-is and is co-located on the dashboard. This phase adds the four healthy-progression phase counts
*alongside* it.

## Color

The dashboard already assigns **one hue per pipeline state** as its established visual language; the admission
cards extend that language with one hue per admission phase, drawing **only on hues already present** in the
dashboard.

| Role | Token | Usage |
|------|-------|-------|
| Dominant (60%) | page background from `base.html` (`white` / dark `phaze` bg) | Page surface — unchanged |
| Secondary (30%) | `bg-gray-50 dark:bg-phaze-panel`, border `border-gray-200 dark:border-phaze-border` | The card `<section>` carrier shell (mirrors `awaiting_cloud_card.html:20`) |
| Accent (10%) | per-phase hues below | The **count tiles only** |
| Reserved fault | `amber` (`bg-amber-50 dark:bg-amber-950`, `text-amber-*`) | **RESERVED for the existing inadmissible alert.** Admission phases MUST NOT use amber — keeping amber meaningful as the genuine "something is wrong" signal. |

### Per-phase semantic color mapping

| `cloud_phase` | Semantic | Tile background | Count / label text | Mirrors existing token |
|---------------|----------|-----------------|--------------------|------------------------|
| `queued_behind_quota` | neutral — waiting for cluster quota | `bg-gray-50 dark:bg-phaze-panel` | `text-gray-700 dark:text-gray-300` | "Executed" tile (`stats_bar.html:22-24`) |
| `admitted` | active — quota granted, pod starting | `bg-blue-50 dark:bg-blue-950` | `text-blue-700 dark:text-blue-400` | "Discovered" tile (`stats_bar.html:2-4`) |
| `running` | active — pod analyzing | `bg-violet-50 dark:bg-violet-950` | `text-violet-700 dark:text-violet-400` | "Analyzing (cloud)" tile (`analyzing_cloud_card.html:21-23`) |
| `finished` | terminal success — result returned | `bg-green-50 dark:bg-green-950` | `text-green-700 dark:text-green-400` | "Approved" tile (`stats_bar.html:18-20`) |

> Divergence from the lead's "e.g. neutral for finished" suggestion: `finished` uses **green**, not a second
> neutral. Two identical-gray tiles (queued + finished) would be visually indistinguishable at a glance; green
> reads as terminal success and is already the dashboard's "done/approved" hue. `queued_behind_quota` remains
> the single neutral. This is a deliberate, documented refinement of the `e.g.` guidance (the lead flagged it as
> an example, not a lock).

Accent reserved for: **the four admission-phase count tiles' background + text only.** Never applied to
interactive controls, headings, borders, or page chrome. Amber is never used by this surface.

## Empty-State & Count Behavior (the locked gating contract)

**Per-tile gating** mirrors `inadmissible_card.html`'s `{% if count %}` (line 21): **a phase with 0 files is
invisible** — its tile is not rendered. Rationale:

- For `a1` and `local` deploys, `cloud_phase` is **always NULL** (Kueue admission is a k8s-only concept —
  RESEARCH Open Q3), so all four counts are 0 → the section renders as an **empty carrier** and the dashboard
  stays quiet. No four-always-zero clutter for non-k8s operators.
- The outer `{% if any count %}` gates the **heading + grid together**, so when all four are 0 nothing visible
  renders — exactly mirroring how the inadmissible alert shows nothing at count 0.
- The empty `<section>` carrier is **always emitted** so the 5s OOB swap has a stable target and a
  previously-shown set **clears** the moment k8s drains (all counts return to 0). This is the same
  carrier-always / body-conditional split the inadmissible card uses.

**Degrade-safe reads:** each count arrives from a `_safe_count`-backed service read
(`get_cloud_phase_counts`, mirroring `get_inadmissible_count`, `pipeline.py:820`). A DB error surfaces as **0**,
so the card renders empty and **never raises into the hot 5s poll** (mirrors T-54-10 / the inadmissible degrade
contract). There is no separate error banner — a degraded read is indistinguishable from "no k8s activity,"
which is the correct quiet failure mode for an observability card.

## Accessibility

| Concern | Contract | Rationale |
|---------|----------|-----------|
| Landmark | `<section id="admission-state-card" aria-labelledby="admission-state-heading">` | Mirrors `awaiting_cloud_card.html:17-21` (labelled section). |
| Heading | `<h2 id="admission-state-heading">` references the section via `aria-labelledby` | Same as all sibling cards. |
| `role="alert"` | **Deliberately OMITTED.** | These are **healthy progression** counts re-rendered every 5s. `role="alert"` (used by `inadmissible_card.html:22`) would make assistive tech announce the counts on every poll tick — noisy and wrong. The sibling observability cards (`awaiting_cloud`, `staged_pushing`, `analyzing_cloud`) all correctly omit it; only the genuine **fault** (inadmissible) is an alert. This is the one attribute where mirroring the inadmissible card would be incorrect; mirror its **structure** (section carrier + aria-labelledby + OOB), not its alert role. |
| Color independence | Each tile carries a **text label + sub-label**, never color alone, so phase identity does not depend on hue (WCAG 1.4.1) | mirrors every sibling tile |
| Contrast | All `text-<hue>-600/700` on `<hue>-50` backgrounds (and `-400` on `-950` dark) reuse the exact pairings already shipped on the dashboard | inherited, pre-validated |
| No live-region churn | Card sits outside `#pipeline-stats`; OOB swap replaces it wholesale without an `aria-live` region | mirrors siblings |

## Copywriting Contract

All strings are **static, autoescaped** Jinja literals — no operator free-text or PII is interpolated (only the
plain-int counts), mirroring T-54-11.

| Element | Copy |
|---------|------|
| Section heading | `Cloud · Admission` |
| `queued_behind_quota` label / sub | `Queued (quota)` / `waiting for cluster quota` |
| `admitted` label / sub | `Admitted` / `quota granted — starting` |
| `running` label / sub | `Running` / `pod analyzing` |
| `finished` label / sub | `Finished` / `result returned` |
| Empty state | **No copy renders.** When all four counts are 0 the section is an empty carrier (no heading, no tiles) — the dashboard stays quiet for a1/local deploys. Mirrors the inadmissible card's silent-at-0 behavior. |
| Error state | **No error banner.** A degraded (`_safe_count` → 0) read is indistinguishable from "no activity" and renders the same empty carrier; the hot poll never surfaces an error to the operator. |
| Primary CTA | **none** — read-only observability surface. (The "Backfill to K8s" action is D-03/KROUTE-05, contracted in the phase plan, not this card.) |
| Destructive confirmation | **none** — no destructive action in this surface. |

> Naming note: `Cloud · Admission` follows the existing `Cloud · Staged` / `Cloud · Analyzing` heading convention
> (`staged_pushing_card.html:20`, `analyzing_cloud_card.html:20`) so the new card slots into the established
> cloud-card family.
