<objective>
Add the ledger-scoped "Backfill to K8s" operator action (D-03 / KROUTE-05) and the static AST guard
(KROUTE-04) that locks the no-over-enqueue property. Depends on Plan 01 (`cloud_target` gate +
`routers/pipeline.py` rekeys already landed). Runs in Wave 2 parallel to Plan 03 (disjoint files).

This plan exists to NOT repeat the v4.0.6 / v5.0 whole-backlog over-enqueue incidents. Two landmines:
- **L4 (scope):** the current `_backfill_candidates_stmt` filters only `ANALYSIS_FAILED ∧ duration ≥
  threshold` — it does NOT require a ledger row. D-03 needs "previously-scheduled work only." Add an
  `EXISTS (scheduling_ledger WHERE key = 'process_file:'||file.id)` predicate.
- **L3 (no seed):** the existing a1 backfill seeds a `process_file:<id>` ledger row per held file.
  For k8s that row would let `recover_orphaned_work` replay the file onto a LOCAL agent queue
  (CLOUDROUTE-02 hazard — the `cloud_job` row, not the ledger, is the k8s in-flight registry). The k8s
  branch resets to DISCOVERED and lets the duration router + `stage_cloud_window` k8s branch pick it up,
  with NO ledger seed.

Open Q2 (RESEARCH): branch the existing `/pipeline/backfill-cloud` endpoint on `cloud_target` (one
surface, mirrors D-01's "one branch") rather than a distinct endpoint — chosen here for the single-seam
philosophy.

Purpose: operator-controlled, bounded, ledger-scoped re-drive of timed-out long files to k8s.
Output: the ledger-scoped candidate query, the k8s backfill fork, the AST-guard extension.
</objective>
