  <behavior>
    - Iteration source (D-02): SELECT cloud_job WHERE status IN (SUBMITTED, RUNNING); each row reconciled against the kube API via kube_staging.get_job(kueue_workload) + get_workload_for(job_uid).
    - Pending (QuotaReserved/False/Pending): silent — no attempts change, no inadmissible flag, no alert; row stays SUBMITTED.
    - Inadmissible (QuotaReserved/False/Inadmissible): set cloud_job.inadmissible=True + WARNING log; attempts UNCHANGED; row held (D-06/D-07).
    - Admitted/QuotaReserved-True: status advances SUBMITTED→RUNNING; no terminal action.
    - Succeeded (Job succeeded>=1 / Complete): record success on cloud_job (SUCCEEDED) and commit, then delete the Job; NO S3 delete here (callback already deleted); reconcile does NOT write an analysis result.
    - Failed/Evicted (no-callback terminal): ORDER record outcome+commit → s3_staging.delete_staged_object(file_id) → kube_staging.delete_job(name). Under cap: increment cloud_job.attempts and re-drive (enqueue a fresh submit_cloud_job for file_id). At cap (attempts+1 > cloud_submit_max_attempts): mark FileRecord ANALYSIS_FAILED, terminate the cloud_job lifecycle (no cross-target fallback).
    - Re-drive race guard: on the under-cap re-drive path, after delete_job the prior Job is confirmed GONE (kube_staging.get_job(name) returns None / NotFound) BEFORE enqueuing the fresh submit_cloud_job. This prevents the Background-GC race where the deterministic-name 409->refresh re-acquires the still-terminating Failed Job, the next tick re-sees Failed, and an extra attempt is burned. (Bounded either way, but the guard keeps the attempt count honest.)
    - Inadmissible NEVER consumes the cap even across many ticks.
    - delete_job is idempotent (a missing Job is fine); the loop never raises on a single bad row (per-row guard, like the cron no-op discipline).
  </behavior>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| kube API status → control plane | reconcile reads kube/Workload status but it is NOT the authoritative result source |
| /api/internal/agent/* callback → control plane | the ONLY authoritative analysis-result channel (reconciled by file_id) |
| reconcile → S3 + kube delete | terminal cleanup mutates external state (object + Job) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-54-15 | Spoofing/Repudiation | trusting kube status as the analysis result | mitigate | KSUBMIT-03: reconcile NEVER calls put_analysis/report_analysis_failed; a grep test asserts zero result-writer calls — the out-of-band callback is the sole authoritative result channel |
| T-54-16 | Tampering | TTL-vs-read race losing a terminal outcome | mitigate | D-04 delete-after-record ordering (record+commit → S3 delete → Job delete); the ordering test proves the Job delete follows the commit; TTL(900s) is the orphan backstop only |
| T-54-17 | Information disclosure | staged S3 object leak on eviction/lost pod | mitigate | D-05: no-callback terminal calls delete_staged_object(file_id) before Job delete; bucket-lifecycle TTL is the backstop |
| T-54-18 | Denial of service | unbounded re-drive loop | mitigate | D-08 bounded cloud_submit_max_attempts → ANALYSIS_FAILED; Inadmissible (operator fault) holds without consuming the cap (D-07) so it never storms |
| T-54-19 | Elevation of privilege | process_file ledger seed via re-drive | mitigate | KSUBMIT-06: the re-drive enqueues submit_cloud_job (controller queue), never a process_file ledger row; reconcile reads cloud_job, never recover_orphaned_work |
</threat_model>
