<objective>
Close the single CR-01 root cause that failed Phase-69 VERIFICATION success criteria 1 and 3:
`LocalBackend.dispatch` enqueues `process_file` but never flips `file.state` out of `AWAITING_CLOUD`
(its compute/kueue siblings flip to `PUSHING` — backends.py:252/:316). Because
`get_cloud_staging_candidates` (pipeline.py:1255) selects `state == AWAITING_CLOUD`, a locally-spilled
file stays a live drain candidate every `*/5` tick and gets re-routed to a cloud backend once a slot
frees — WHILE its local `process_file` runs. That is a cross-backend double-analysis (different SAQ job
keys, no dedup) plus a stranded `SUBMITTED` compute `cloud_job` (compute reconcile is a no-op) = leaked cap slot.

Fix: `LocalBackend.dispatch` atomically flips `file.state` into a NEW dedicated in-analysis state
(`FileState.LOCAL_ANALYZING`) in the SAME session as the enqueue, mirroring the compute/kueue `PUSHING`
flip and the drain's single post-loop commit. Fold in the co-located WR-01 return-value honesty fix.

Purpose: make local a genuine drain-terminal target so the tiered scheduler's rank-first exclusivity
(SCHED-01) and black-hole/local-terminal guard (SCHED-03) actually hold for the local leg.
Output: `FileState.LOCAL_ANALYZING`; a corrected `LocalBackend.dispatch`; RED→GREEN regression tests.

## FileState decision (RESOLVED — justified against the 4 required constraints)

**Chosen value: `LOCAL_ANALYZING = "local_analyzing"`** — a NEW dedicated in-analysis state.

- Length: `"local_analyzing"` is 15 chars ≤ `String(30)` — code-only StrEnum over the existing
  `files.state` column, NO migration (exactly the `AWAITING_CLOUD` / `ANALYSIS_FAILED` precedent
  documented in models/file.py:37-50).
- Constraint 1 (excluded from `get_cloud_staging_candidates`): satisfied — it is NOT `AWAITING_CLOUD`,
  so the candidate predicate (`state == AWAITING_CLOUD`) no longer matches → no re-selection.
- Constraint 2 (NOT analyze-done, NOT push-done, so a lost local job re-drives): satisfied — it is NOT
  in `_select_done_analyze_ids` `{ANALYZED, ANALYSIS_FAILED}` (reenqueue.py:179) NOR `_select_done_push_ids`
  `{PUSHED, ANALYZED, ANALYSIS_FAILED}` (reenqueue.py:189). If the local agent dies mid-analysis, the
  `process_file` scheduling-ledger row is still un-cleared and `is_domain_completed` returns False →
  the ledger re-drives it (a lost local job stays recoverable, not falsely "done"). It also carries NO
  `cloud_job` row, so `_in_flight_cloud_job_ids` (reenqueue.py:204-219) does NOT exclude it from recovery.
- Constraint 3 (NOT PUSHING/PUSHED): satisfied — those imply a `cloud_job` row + compute `/pushed`/reconcile
  ownership the local path does not have; a distinct state prevents reconcile/recovery misclassifying a
  local file as cloud-owned.
- Constraint 4 (NOT FINGERPRINTED): satisfied — a distinct state cannot be re-routed to AWAITING_CLOUD by
  fingerprint-completion routing and does not re-open the normal-producer double-enqueue window.
- Completion path confirmed: `put_analysis` (agent_analysis.py:221-222) flips `state = ANALYZED` gated
  ONLY on a non-empty result body (`dumped`), NEVER on prior state, so `LOCAL_ANALYZING → ANALYZED`
  completes normally.

## Out of scope (tracked, NOT dropped — do NOT pull into this gap plan)

- **WR-02** (config.py `_validate_registry`): no whole-registry invariant requiring ≥1 `kind == "local"`
  backend when `cloud_enabled`; a cloud-only registry can wedge a cloud-exhausted file in AWAITING_CLOUD
  permanently. Pre-existing / Phase-70 registry-validation territory.
- **WR-03** (compute reconcile no-op + `_in_flight_cloud_job_ids` exclusion): a stuck compute `PUSHING`
  row whose `push_file` job was lost has ZERO recovery owner. Broader robustness; Phase-70 territory.

Both are explicitly deferred here so re-verification does not treat them as newly-introduced regressions.
