<objective>
Turn the single-backend drain into the tiered multi-backend scheduler. Remove the Phase-68 `>1 non-local` boot guards (supporting N non-local backends is this phase's job), then rewire `stage_cloud_window` to (a) snapshot every backend's `is_available()` + `remaining = cap - in_flight_count()` ONCE per tick under the existing advisory lock, (b) select each FIFO candidate's target via the pure `select_backend` (Plan 01), (c) `dispatch()` + decrement the local `remaining[]`, all in one transaction with a single post-loop commit. Retire the global `get_cloud_window_count()` window in favor of per-backend `in_flight_count()`.

Purpose: This is the moment more than one backend can run at once (SCHED-01) and the moment per-backend `cap` is enforced by count-and-claim under the lock (SCHED-02).
Output: multi-backend drain + the extended drain/backends test suites.
</objective>
