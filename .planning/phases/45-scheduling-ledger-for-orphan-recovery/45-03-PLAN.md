<objective>
Rewrite `recover_orphaned_work` to drive recovery off the durable ledger instead of the
complement-of-done pending-set queries. Recovery re-enqueues exactly
`ledger − live-saq_jobs-keys − domain-completed`, replaying each orphaned row's STORED payload
through the SAME keyed producer that originally enqueued it. This is the fix: a never-scheduled
DISCOVERED file has no ledger row, so the ~11.4k-file queue detonation cannot recur.

Purpose: "Recovery only re-queues work that was previously scheduled and then lost" (the operator
spec). The ledger is that record. Live keys (queued/active) are excluded (still in flight); the
domain-completed predicate is the SECONDARY net for the documented residual gap (Plan 02: a
metadata/fingerprint job that exhausts retries without a callback). force=True flips to "reconcile
the ledger now", bypassing only the no-op gate, never the per-item deterministic-key dedup.

Output: rewritten reenqueue.py recovery producer + recovery tests seeded from ledger rows.

## Per-stage domain-completed classification (Plan-checker Blocker 2 / W-note resolution)

FileState is a SINGLE column with NO per-stage FAILED states, so a FileState-derived "done"
predicate is reliable only where a stage's success advances the column past its own gate. This plan
makes the exclusion EXPLICIT and TESTED per stage — no stage is left silently undefined:

| Stage (ledger function) | Routing | Domain-completed predicate (SECONDARY exclusion) | Why |
|-------------------------|---------|--------------------------------------------------|-----|
| process_file (analyze) | agent | FileRecord.state in {ANALYZED, ANALYSIS_FAILED} | analyze has BOTH a success PUT and a /failed callback (Plan 02) that clear the ledger directly; the predicate is a belt-and-suspenders net for a crash-without-callback. |
| extract_file_metadata | agent | FileRecord.state advanced past METADATA_EXTRACTED gate (the existing get_metadata_pending_files boundary — a file NOT in the pending set is done) | NO /failed callback (Plan 02 residual gap); this predicate is the PRIMARY net for a retries-exhausted-no-callback metadata job. |
| fingerprint_file | agent | file has a FingerprintResult row (the existing get_fingerprint_pending_files boundary — not in pending ⇒ done) | NO /failed callback (Plan 02 residual gap); PRIMARY net for fingerprint. |
| scan_live_set | agent | (none needed — LIVE-KEYS-ONLY) | Plan 02's terminal-ack clears scan_live_set:<file_id> on EVERY outcome (match/no-match/failure), so the ledger row is already gone; no domain predicate required. The earlier no-match hole is closed at the source. |
| generate_proposals | controller | (none needed — LIVE-KEYS-ONLY) | controller stages clear via Plan 01's after_process on terminal status; no callback gap. |
| search_tracklist | controller | (none needed — LIVE-KEYS-ONLY) | same as above. |
| scrape_and_store_tracklist | controller | (none needed — LIVE-KEYS-ONLY) | same as above. |
| match_tracklist_to_discogs | controller | (none needed — LIVE-KEYS-ONLY) | same as above. |

So the domain-completed predicate is applied to EXACTLY THREE agent stages (process_file,
extract_file_metadata, fingerprint_file); scan_live_set and the four controller stages are
live-keys-only because their ledger clear is reliable on every terminal outcome. This enumeration is
asserted in tests (a parametrized "each function is predicate-covered XOR live-keys-only" check).
</objective>
