<objective>
Clear AGENT-STAGE ledger entries on the CONTROL side, in the HTTP callbacks the agent already
invokes (plus ONE new terminal-ack endpoint for scan's no-match/failure hole). This resolves the
phase's headline structural risk: the four agent stages run their SAQ `after_process` on the
Postgres-free agent worker, which CANNOT clear a control-side ledger (Plan 01's after_process clear
only reaches Postgres on the controller worker). The earliest control-visible moment an agent job's
outcome is known is its callback to the API — so that is where the agent-stage ledger row is cleared.

Purpose: Honor locked decision #1 (terminal `failed` must NOT recovery-re-queue — the poison case)
for the agent stages, and clear on EVERY terminal outcome (success, no-match, failure) so recovery
never re-runs finished work and never loops forever on a legitimate no-match scan.

Output: ledger clears wired into the existing agent-callback handlers, a new scan terminal-ack
endpoint + agent-client method, the scan-task wiring, and tests.

## HEADLINE DECISION — agent-stage terminal clear (RESOLVED)

Verified callback surface for the four agent stages (this phase, against the codebase):

| Agent stage | Ledger key | Success callback (clears here) | Terminal-FAILURE / no-result clear |
|-------------|-----------|--------------------------------|------------------------------------|
| analyze / process_file | process_file:<file_id> | PUT /analysis/{file_id} (sets ANALYZED) | YES — POST /analysis/{file_id}/failed (sets ANALYSIS_FAILED; emitted by the task's timeout/crash/terminal-error paths, functions.py:173/177/185). **Poison-case clear — decision #1 honored.** |
| extract_file_metadata | extract_file_metadata:<file_id> | PUT /metadata/{file_id} | NO dedicated /failed callback. Residual gap (see below) — covered by Plan 03's domain-completed exclusion; metadata failure is transient I/O, SHOULD re-attempt on recovery (not the poison case). |
| fingerprint_file | fingerprint_file:<file_id> (single key per file, NOT per engine) | PUT /fingerprints/{file_id}/{engine} | NO dedicated /failed callback. Same residual-gap treatment as metadata. |
| scan_live_set | scan_live_set:<file_id> | create_tracklist (POST /tracklists; body carries file_id) ON A MATCH | **NEW: POST /tracklists/{file_id}/scanned terminal-ack** covers the no-match early-return AND the terminal-failure path. (Plan-checker Blocker 1+2 fix — see below.) |

**Plan-checker corrections folded in:**
- **Blocker 1:** the scan clear is NOT in `agent_scan_batches.patch_scan_batch`. `ScanBatch`
  (models/scan_batch.py) has NO `file_id` column (verified — it is keyed by agent_id/scan_path), and
  that endpoint is the callback for the DISCOVERY stage `scan_directory` (NOT in _KEY_BUILDERS) —
  unrelated to `scan_live_set`. The correct surfaces carry `file_id`:
  `agent_tracklists.create_tracklist` (`TracklistCreatePayload.file_id`, schemas/agent_tracklists.py:40)
  for the match path, and a new terminal-ack endpoint on the SAME tracklists router for the rest.
  `agent_scan_batches.py` is NOT touched by this plan.
- **Blocker 2:** `scan_live_set` (tasks/scan.py:95-96) returns early `{"status": "no_matches"}` WITHOUT
  calling create_tracklist when `orchestrator.combined_query` finds nothing — a clean SAQ COMPLETE that
  posts no tracklist. With a tracklist-existence-only exclusion, that no-match row would re-enqueue on
  EVERY recovery (a smaller repeat of the incident class). **Resolution: Option (a)** — add a terminal-ack
  endpoint and have the scan task call it on BOTH the no-match return AND the retries-exhausted terminal
  failure, so every scan run clears scan_live_set:<file_id> EXACTLY ONCE regardless of outcome. Chosen
  over Option (b) (a FileState-derived per-stage predicate) because FileState is a SINGLE column with no
  per-stage FAILED states, making a "scan-done" predicate read from it muddy once a file advances; a
  dedicated ack is unambiguous and self-clearing.

**Residual gap (documented, accepted):** a metadata/fingerprint job that exhausts retries WITHOUT
any callback leaves its ledger row uncleared and is re-attempted on the next recovery. This degrades
to "recoverable" (benign — these are transient I/O failures, not the poison auth/connect case), and
is the secondary safety net in Plan 03's domain-completed exclusion. The incident's actual poison
jobs were analyze (essentia/timeout → /failed, cleared) and the controller proposal/auth-connect
failures (cleared via Plan 01's after_process). Decision #1 is honored for every poison case. If a
future incident shows metadata/fingerprint poison re-queuing, add their /failed callbacks then (the
ledger model + clear helper already support it).

| agent worker -> control API callback -> Postgres ledger | the agent's terminal/success outcome becomes control-visible only here; the ledger clear rides the existing result-write transaction (or the new ack endpoint) |
