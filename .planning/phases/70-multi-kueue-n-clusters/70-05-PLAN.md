<objective>
Clean-before-flip spillover cleanup (MKUE-04, the crux — D-01/D-03): reorder the at-cap spill-back in
`reconcile_cloud_jobs._handle_no_callback_terminal` so the old `(backend_id, staging_bucket)` staged
object is deleted WHILE the per-row `pg_advisory_xact_lock(5_000_504)` is still held, BEFORE the commit
that flips the file to `AWAITING_CLOUD` (and thus releases the lock and makes the file a drain candidate).
Today the code commits first and deletes after — in a multi-bucket world that opens Pitfall 9: a
concurrent drain tick can re-dispatch the file and re-stage a NEW object under the same `file_id`-scoped
key, and if D-06 lands the re-dispatch on the same bucket, the trailing delete destroys the object the
new pod needs. Deleting under the lock, before the flip, guarantees the old object is gone before any
re-stage can occur.

Purpose: a spillover re-dispatch never deletes an object another cluster/bucket still uses; the
per-bucket lifecycle TTL stays a pure backstop.
Output: the reordered at-cap branch + a clean-before-flip ordering test + a same-bucket-re-dispatch
preservation test + a drain↔reconcile concurrency test.

DEPENDS on Plan 02 (`staging_bucket` recorded + `delete_staged_object(file_id, bucket)` + the interim
bucket-aware at-cap delete + `_resolve_bucket_config`) and Plan 03 (reconcile threads the backend's kube).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| reconcile txn ↔ concurrent drain tick | both take pg_advisory_xact_lock(5_000_504); the spillover cleanup correctness lives in the Postgres lock/txn boundary |
| control plane → S3 (old + new bucket) | the delete targets the RECORDED (backend, bucket); the file_id-scoped key is shared across re-dispatches |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-70-04 | Tampering / Denial of Service | cross-bucket object destruction on spillover (a new pod's object deleted) | mitigate | clean-before-flip: delete the old (backend, RECORDED staging_bucket) object UNDER the held advisory lock, BEFORE the AWAITING_CLOUD commit; the drain cannot claim the file until reconcile commits (Pitfall 9) |
| T-70-04-02 | Denial of Service | a slow/failed S3 delete pinning the per-row lock | mitigate | delete is idempotent + `contextlib.suppress(Exception)`-wrapped (D-03), so it releases within one network timeout; per-bucket TTL is the backstop; reconcile is a background */5 cron, not the latency-sensitive dispatch path |
| T-70-04-03 | Tampering | whole-tick lock breaking per-row commit granularity | mitigate | keep the lock PER-ROW (Pitfall 2); the delete runs inside the existing per-row unit, not a widened lock |
| T-70-04-04 | Information Disclosure | recorded-bucket mis-resolution | mitigate | read the RECORDED cloud_job.staging_bucket (captured pre-mutation), never re-derive; clear it to None after cleanup so no pre-repurpose reader is misled |
</threat_model>
