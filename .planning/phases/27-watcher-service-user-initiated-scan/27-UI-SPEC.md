# Phase 27 — UI Design Contract

> Visual and interaction contract for the Trigger Scan card, Scan Progress poll partial, Recent Scans mini-table, and Agent → Scan-Root HTMX swap. Generated by gsd-ui-researcher.

Phase 27 extends the existing operator-facing `/pipeline/` dashboard. **All tokens below are INHERITED from the existing Phaze design system** (Tailwind v4 + custom theme in `templates/base.html` + established class patterns across `pipeline/`, `tracklists/`, `execution/`, `proposals/` templates). Phase 27 introduces **no new visual tokens**; it composes existing ones.

---

## Design System

| Property | Value |
|----------|-------|

## Typography

Inherited from `templates/base.html` (`body { font-family: 'Inter', sans-serif }`) and the established usage across `templates/pipeline/`, `templates/execution/`, `templates/proposals/`. Phase 27 declares **3 sizes + 2 weights**.

| Role | Tailwind class | Size | Weight | Line Height | Usage in Phase 27 |
|------|----------------|------|--------|-------------|--------------------|
| Page title | `text-2xl font-semibold leading-tight` | 24px | 600 | 1.25 | Reuses existing `<h1>Pipeline Dashboard</h1>` — Phase 27 does NOT add an `<h1>` |
| Card title | `text-lg font-semibold` | 18px | 600 | 1.55 (Tailwind default) | "Trigger Scan", "Recent Scans", "Scan Progress" card headings (`<h2>`) |
| Body / form label / table cell | `text-sm` | 14px | 400 (form values) / 600 (form labels) | 1.43 (Tailwind default) | Form labels, helper text, table cell text, status text, error messages |
| Helper / micro-label / table header | `text-xs font-semibold uppercase` | 12px | 600 | 1.5 (Tailwind default) | Table column headers (`<th>`), micro-stats labels, status pill text |

**Weights used:** 400 (Inter regular, default body) + 600 (Inter semibold, all headings + labels + emphasis). No 300, 500, or 700 weights in Phase 27 UI.

**Font family overrides:** none. Phase 27 does NOT use the `Jura` display font (reserved for the `PHAZE` brand mark in the nav).

**Letter-spacing:** none beyond Tailwind defaults. The `tracking-[0.2em]` value in `base.html` is reserved for the brand mark.

- Table row hovers (those are secondary `bg-gray-50 dark:bg-phaze-panel`)
- Status pills for running scans (those use `blue-100/950` not `blue-600/700` — the surface variant of the same hue is the established "info pill" pattern, see `status_badge.html`)
- Secondary action buttons (Phase 27 ships no secondary action buttons)

**Destructive reserved for:** the FAILED status pill, the error-message paragraph beneath a failed scan row in Recent Scans, and the tooltip body when hovering a failed row.

**Contrast verification (WCAG 2.1 AA, 4.5:1 minimum for normal text):**

| Pair | Ratio | Pass |
|------|-------|------|
| `text-gray-900` (#111827) on `bg-white` | 17.86:1 | yes |
| `text-gray-100` (#f3f4f6) on `bg-phaze-bg` (#0a0c12) | 16.96:1 | yes |
| `text-gray-500` (#6b7280) on `bg-white` (helper text) | 4.83:1 | yes (AA normal) |
| `text-gray-400` (#9ca3af) on `bg-phaze-bg` (#0a0c12) (helper text, dark) | 6.99:1 | yes |
| `text-white` on `bg-blue-600` (#00b0d8) (button label) | 3.04:1 | **borderline** — see note below |
| `text-red-700` (#b91c1c) on `bg-red-100` (#fee2e2) (failed pill) | 6.31:1 | yes |
| `text-green-700` on `bg-green-100` (completed pill) | 5.78:1 | yes |
| `text-blue-700` on `bg-blue-100` (running pill) | 5.13:1 | yes |

**Note on `text-white` on `bg-blue-600`:** Phaze's custom `#00b0d8` is a lighter cyan than Tailwind's default `#2563eb`, producing a borderline 3.04:1 contrast for normal text. **For Phase 27's submit button, the button text uses `text-sm font-semibold` (14px @ weight 600) which under WCAG counts as "large bold text" requiring only 3:1.** Pass at 3.04:1, but tight — checker should re-verify if the rendered text shrinks. This matches the existing `bg-blue-600 text-white` button pattern in `stage_cards.html` line 19, so Phase 27 inherits the same risk as the rest of the app and does NOT diverge here.

## Component Contracts

### Component 1: Trigger Scan card

**Container:**
```html
<section aria-labelledby="trigger-scan-heading"
         class="border border-gray-200 dark:border-phaze-border rounded-lg p-4 space-y-4">
  <h2 id="trigger-scan-heading" class="text-lg font-semibold">Trigger Scan</h2>
  <p class="text-sm text-gray-500 dark:text-gray-400">
    Walk a directory on a registered agent and ingest discovered files.
  </p>
  <form ...> ... </form>
  <div id="scan-submit-result" aria-live="polite"></div>
</section>
```

**Form layout** (vertical stack, full-width on desktop, fields stack inside `space-y-4`):

| Order | Field | Type | Required | Behavior |
|-------|-------|------|----------|----------|
| 1 | Agent | `<select id="scan-agent" name="agent_id">` | yes | On `change` (HTMX trigger), fetches `GET /pipeline/scans/agent/{agent_id}/scan-roots` and swaps `#scan-path-picker`. Default option `<option value="" disabled selected>Select an agent…</option>` |
| 2 | Scan root | `<select id="scan-root" name="scan_root">` (rendered inside `#scan-path-picker`) | yes | Populated by HTMX swap. Pre-swap (no agent picked): shows disabled placeholder `<option value="" disabled selected>Select an agent first</option>`. Post-swap: one `<option>` per entry in `agent.scan_roots`, plus the same disabled placeholder at the top |
| 3 | Subpath (optional) | `<input id="scan-subpath" name="subpath" type="text" placeholder="e.g. 2026-coachella/">` | no | Plain text input; leading `/` allowed but stripped server-side. **Placeholder is conservative-empty per CONTEXT D-Discretion ruling — no pre-population of `scan_root + /`** |
| 4 | Submit | `<button type="submit">Start Scan</button>` | — | Primary CTA; `bg-blue-600 dark:bg-blue-700 text-white` per accent contract |

**Field markup pattern** (mirrors form patterns from `templates/search/partials/search_form.html`):

```html
<div>
  <label for="scan-agent" class="block text-sm font-semibold text-gray-700 dark:text-gray-300 mb-1">
    Agent
  </label>
  <select id="scan-agent" name="agent_id" required
          hx-get="/pipeline/scans/agent-roots"
          hx-target="#scan-path-picker"
          hx-swap="innerHTML"
          hx-trigger="change"
          hx-include="this"
          class="w-full text-sm rounded-md border border-gray-300 dark:border-phaze-border bg-white dark:bg-phaze-bg text-gray-900 dark:text-gray-100 px-3 py-2 focus:outline-none focus:ring-2 focus:ring-blue-500 focus:border-blue-500">
    <option value="" disabled selected>Select an agent…</option>
    {% for agent in agents %}
    <option value="{{ agent.id }}">{{ agent.name }} ({{ agent.id }})</option>
    {% endfor %}
  </select>
</div>
```

**Agent option label format:** `{name} ({id})` per CONTEXT D-Discretion ruling. Aids debugging when multiple agents share similar names.

**Scan-Path Picker slot:** `<div id="scan-path-picker" class="space-y-4">…</div>`. The HTMX swap replaces innerHTML with the rendered partial `pipeline/partials/scan_path_picker.html`. The partial contains BOTH the scan_root `<select>` AND the subpath `<input>` so the operator sees the constrained options immediately on agent selection.

**Pre-selection state** (rendered on first page load, no agent picked yet):
```html
<div>
  <label for="scan-root" class="block text-sm font-semibold text-gray-500 dark:text-gray-400 mb-1">
    Scan root
  </label>
  <select id="scan-root" disabled
          class="w-full text-sm rounded-md border border-gray-200 dark:border-phaze-border bg-gray-50 dark:bg-phaze-panel text-gray-400 dark:text-gray-500 px-3 py-2 cursor-not-allowed">
    <option>Select an agent first</option>
  </select>
</div>
```

**Submit button:**
```html
<button type="submit"
        class="px-4 py-2 bg-blue-600 dark:bg-blue-700 text-white rounded-md hover:bg-blue-700 dark:hover:bg-blue-600 disabled:opacity-50 disabled:cursor-not-allowed text-sm font-semibold focus:outline-none focus:ring-2 focus:ring-blue-500 focus:ring-offset-2 dark:focus:ring-offset-phaze-bg">
  Start Scan
</button>
<span id="scan-submit-spinner" class="htmx-indicator ml-2 text-sm text-gray-500 dark:text-gray-400">Enqueuing…</span>
```

The button uses the same indicator pattern as `stage_cards.html`. Disabled when `scan-agent` or `scan-root` has no value (Alpine.js handles this via `x-data="{ agentId: '', scanRoot: '' }"` + `:disabled="!agentId || !scanRoot"`).

**Container:**
```html
<section aria-labelledby="recent-scans-heading"
```

```html
<div class="text-center py-8">
```

**Row count:** last 10 ScanBatches across all agents, sorted `created_at DESC`. Only `running`, `completed`, `failed` rows appear.

**Column widths:** Path column is the only one that can overflow; uses `truncate max-w-md` with a `title=` tooltip for full path. Other columns use natural widths.

## Interaction Contracts

### Form submit → optimistic UI (CONTEXT D-08 + Specifics §6)

1. Operator picks agent → HTMX swap replaces `#scan-path-picker` with the agent's scan-root `<select>` + subpath input.
2. Operator picks scan root + optional subpath → Submit button enables.
3. Operator clicks "Start Scan" → form `hx-post="/pipeline/scans"` with `hx-target="#scan-submit-result"` `hx-swap="innerHTML"` `hx-indicator="#scan-submit-spinner"`.
4. Controller validates `(agent_id, scan_root, subpath)`, creates `ScanBatch(status=RUNNING)`, enqueues `scan_directory` via `AgentTaskRouter`, returns rendered `scan_progress_card.html` (in-progress state) into `#scan-submit-result`.
5. **The form itself stays open** — fields are NOT cleared, allowing the operator to trigger another scan immediately by changing the agent/path and re-submitting.
6. The progress card begins polling `GET /pipeline/scans/{batch_id}` every 2s.
7. When the polled response renders a terminal state (no `hx-trigger`), polling stops.

**Submit failure surfacing:**
- 400 validation error (subpath contains `..`, scan_root not in agent.scan_roots, or empty subpath that resolves to root): controller returns rendered `pipeline/partials/scan_submit_error.html` into `#scan-submit-result`:
  ```html
  <div class="bg-red-50 dark:bg-red-950/30 border border-red-200 dark:border-red-900 rounded-md p-3" role="alert">
    <p class="text-sm text-red-700 dark:text-red-400">{{ error_message }}</p>
  </div>
  ```
- 403 cross-tenant (impossible from UI — admin owns all agents — but defensive): same shape, copy = `You do not have permission to trigger a scan on this agent.`
- 5xx server error: same shape, copy = `The application server could not enqueue the scan. Try again in a moment.`

### Polling cadence (CONTEXT D-08)

| Surface | Endpoint | Cadence | Halt condition |
|---------|----------|---------|----------------|
| Scan Progress card | `GET /pipeline/scans/{batch_id}` | every 2s | response renders terminal state (no `hx-trigger` in returned markup) |
| Pipeline stats bar | `GET /pipeline/stats` | every 5s | none (page lifetime) |
| Recent Scans table | (rides stats poll via OOB swap) | every 5s | none (page lifetime) |

**No SSE in Phase 27.** SSE is reserved for Phase 28 execution-dispatch aggregation.

|-----|---------|--------|

| `Shift+Tab` | anywhere | reverse order |

**Focus management on terminal-state swap:** When the Scan Progress card transitions from in-progress to a terminal state, focus is NOT moved (the operator may be elsewhere on the page). **Focus management on successful form submit:** focus stays on the Start Scan button (HTMX does not move focus by default and Phase 27 does not override). The `aria-live="polite"` region inside `#scan-submit-result` announces the new progress card.

## Accessibility

### ARIA & semantic HTML

| Concern | Solution |
|---------|----------|
| Live region for poll partial | `aria-live="polite"` on the Scan Progress card outer `<div>` and on `#scan-submit-result` |
| Live region for error message swap | Inherits from `#scan-submit-result`'s `aria-live="polite"` |
| Form field labels | Every input/select has a visible `<label for=...>`. Placeholder text is NEVER the only label (per WCAG 2.1 §3.3.2). |
| Status pill not color-only | Each pill renders the text `RUNNING` / `COMPLETED` / `FAILED`. `aria-label="Status: <state>"` is set as a redundant cue. |
| Table semantics | `<table>` with proper `<thead>` / `<tbody>` / `<th scope="col">`. Row hover is non-interactive (no `<button>` or `<a>` inside rows in Phase 27). |
| Error alert region | `role="alert"` on the scan-submit-error card (announces immediately, not just on focus) |
| Focus indicators | Tailwind's `focus:ring-2 focus:ring-blue-500 focus:ring-offset-2 dark:focus:ring-offset-phaze-bg` on every interactive element — never `focus:outline-none` without a replacement ring |
| Section landmarks | Each card is a `<section>` with `aria-labelledby=` pointing at its `<h2>` |
| Skip link | The existing `base.html` skip link is dashboard-targeted (`#pipeline-stages`). Phase 27 does NOT add a competing skip-link target; the Trigger Scan card is the first interactive region after nav and is reachable by 1–2 Tab presses. |

### Contrast (recap)

All foreground/background pairs meet WCAG 2.1 AA. The only borderline case is `text-white` on Phaze's custom `bg-blue-600` (#00b0d8) at 3.04:1, which passes for "large bold text" given the 14px @ weight 600 button label. This matches the existing app-wide pattern and is NOT a Phase 27 regression.

### Reduced motion

Phaze does not currently respect `prefers-reduced-motion`. Phase 27 introduces no new animations — the `htmx-indicator` opacity transition (`transition: opacity 200ms`) is already in `base.html` and is the only motion in Phase 27. **No new motion to gate.**
