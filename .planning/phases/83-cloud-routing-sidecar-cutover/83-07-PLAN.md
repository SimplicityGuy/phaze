<objective>
Close the single Phase-83 verification gap (WR-03 / failed must-have #7): the LOCKED **D-02:** constraint —
"one writer, reused by the hold path and both spill paths — not three hand-written copies" — is not realized
in shipped code. `hold_awaiting_cloud()` (`services/backends.py:84`) has exactly ONE caller
(`trigger_analysis`, `pipeline.py:351`); both over-cap spill paths re-implement their own inline CAS
`UPDATE cloud_job SET status='awaiting' ...` (`agent_s3.py:206-215`, `agent_push.py:273-283`).

This plan consolidates all three awaiting writes into the single `hold_awaiting_cloud` helper by extending it
to a **CAS-preserving dual-mode** signature, then routing both spill paths through it. The refactor is
**behavior-preserving**: every existing green test (the `agents` bucket, `test_drain_double_dispatch.py`,
`test_shadow_compare.py`, `test_backends.py`) stays green, and a new anti-drift source test makes the
single-writer invariant self-enforcing so D-02 cannot silently drift again.

⚠ CRITICAL — the naive fix is FORBIDDEN. The verification's `missing:` field literally suggests routing the
spills through the *existing* unconditional-upsert helper. Taken literally that RE-OPENS the security bug
Phase 83 just closed (`T-83-PUSH-CLOBBER` / SC#2): the existing helper is a `pg_insert(...).on_conflict_do_update(...)`
that ALWAYS writes an awaiting row, destroying the rowcount-guarded CAS whose `rowcount == 0` branch is the
FULL no-op that prevents a late/duplicate callback from clobbering an already-advanced (RUNNING/SUCCEEDED/reaped)
file back to AWAITING_CLOUD. The helper MUST gain a conditional CAS mode instead — the exact operator-approved
shape below.

Purpose: realize the LOCKED D-02 single-writer invariant in code and make it drift-proof, without weakening
the D-09/D-10/D-11/D-12 CAS guards Phase 83 shipped.
Output: extended `hold_awaiting_cloud`, two routed spill call sites, one anti-drift test, helper unit tests.
</objective>
