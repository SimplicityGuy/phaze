<objective>
Land the ExecutionLog router ‚Äî a multi-verb endpoint (POST + PATCH) with an application-level monotonic-status invariant (D-15). This is the most behaviorally-rich of Phase 25's routers because:

1. **Agent-generated UUIDs** (D-13): the agent supplies the `ExecutionLog.id` in the POST body. Server uses `INSERT ... ON CONFLICT (id) DO NOTHING` so retries are no-ops. Agent persists the id in SAQ job state so subsequent PATCHes target the same row across retries.

2. **Monotonic status check** (D-15): the four-state ladder `PENDING < IN_PROGRESS < COMPLETED < FAILED` is enforced at PATCH time. A regression returns 409 with detail `"execution-log status would regress"`. A PATCH against a terminal state (COMPLETED or FAILED) returns 409 with `"execution-log status is terminal"`. Same-status PATCH (idempotent retry) is allowed.

3. **No auto-enqueue** ‚Äî unlike Plan 03's files endpoint, ExecutionLog writes are pure audit-trail entries.

    # `<` (not `<=`) means same-status PATCH is allowed for idempotent retry.
    _STATUS_ORDER: dict[ExecutionStatus, int] = {
        ExecutionStatus.PENDING: 0,
        ExecutionStatus.IN_PROGRESS: 1,

        ExecutionStatus.FAILED: 3,

    ) -> ExecutionLogCreateResponse:
        """Create an ExecutionLog row. Agent supplies `id` (D-13); replay POST = no-op.

        Per D-13: INSERT ... ON CONFLICT (id) DO NOTHING -- first-create wins,
        identical replays are silent no-ops. The agent persists `id` in SAQ job
        state so retries point at the same row.

        stmt = pg_insert(ExecutionLog).values([payload]).on_conflict_do_nothing(index_elements=["id"])
        await session.execute(stmt)

        execution_log_id: uuid.UUID,
        body: ExecutionLogPatch,
        agent: Annotated[Agent, Depends(get_authenticated_agent)],
        session: Annotated[AsyncSession, Depends(get_session)],
    ) -> ExecutionLogPatchResponse:
        """Update an ExecutionLog row. Status transitions are monotonic (D-15).

        - 404 if `execution_log_id` does not exist
        - 409 if current status is terminal (COMPLETED or FAILED)
        - 409 if proposed status regresses (e.g., IN_PROGRESS -> PENDING)

        if existing is None:
            raise HTTPException(status_code=status.HTTP_404_NOT_FOUND, detail="execution-log not found")

        # `existing.status` is stored as String(20); cast back to enum for ordered comparison.
        cur = ExecutionStatus(existing.status)
        new = body.status  # already an ExecutionStatus instance (Pydantic-validated)

        # D-15: terminal-state guard runs FIRST (early exit before regress check)
        if cur in _TERMINAL:

        # D-15: monotonic guard -- `<` (not `<=`) so same-status retry is allowed.
        if _STATUS_ORDER[new] < _STATUS_ORDER[cur]:

        for field, value in body.model_dump(exclude_unset=True).items():
            setattr(existing, field, value)
        await session.commit()
        return ExecutionLogPatchResponse(
            agent_id=agent.id,
            execution_log_id=execution_log_id,
            status=ExecutionStatus(existing.status),
        )
