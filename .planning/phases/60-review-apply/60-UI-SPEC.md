# Phase 60 — UI Design Contract

> Visual and interaction contract for the **Review & Apply gate** — the unified approval surface that
> supersedes SIX `_STAGE_PLACEHOLDER` rail-node workspaces (`propose` · `rename` · `tagwrite` · `move` ·
> `dedupe` · `cue`) in `src/phaze/routers/shell.py:88-93`. Generated by gsd-ui-researcher.
>
> **Authority & inheritance:** This contract **INHERITS `.planning/phases/59-identify-workspaces/59-UI-SPEC.md`
> verbatim** (which inherits 58, which inherits 57). Spacing, typography, color tokens, the shell chrome,
> the DAG rail, the header status strip, the ⌘K modal, the OOB single-poll fanout, `$store.pipeline`, the
> focus/ARIA baseline, AND the inherited workspace patterns (Workspace Scaffold `_workspace_scaffold.html`,
> File Table `_file_table.html`, Lane Card `_lane_card.html`, workspace poll seeds
> `_workspace_poll_seeds.html`) are all **locked upstream and are NOT re-derived here.** This spec
> specifies ONLY what is NEW for the Review & Apply gate: the unified **before→after diff partial**, the
> **bulk high-confidence header action**, the **Dedupe keeper-group**, the **Cue preview card**, and the
> **applied/reversible row state**.
>
> The canonical visual/behavioral target is `docs/superpowers/specs/2026-06-28-ui-redesign-dag-console-design.md`
> **§7 (Review & Apply)** and `docs/superpowers/specs/2026-06-28-ui-redesign-assets/prototype.html` (the
> `diffRow()` / `dupeGroup()` / `propose()` / `rename()` / `tagwrite()` / `move()` / `dedupe()` / `cue()`
> functions). When the prototype's raw mockup hexes / helper classes (`.del`, `.add`, `bg-pp`, `bg-ps`,
> `.bd`) conflict with the production C3 tokens (`phaze-bg` / `phaze-panel` / `phaze-border`,
> `blue-500` / `emerald` / `amber` / `rose`), **the production tokens win** — "evolution, not reskin."

This spec covers the **content of six Review & Apply stage workspaces** that swap into the Phase-57
`#stage-workspace` target, superseding the `propose` / `rename` / `tagwrite` / `move` / `dedupe` / `cue`
`_STAGE_PLACEHOLDER` entries in `shell.py:88-93` (supersede-in-place; legacy templates stay until
CUT-02/Phase 62). It does NOT re-specify the shell chrome or the inherited workspace patterns.

| In scope (contract below) | Out of scope (deferred — note only) |
|---------------------------|--------------------------------------|
| **Unified before→after diff partial** — ONE Jinja partial reused across Rename/Path · Tag-write · Move-files with per-row **Approve / Edit / Skip** (REVIEW-01, D-06) | The shell chrome (header, rail, ⌘K, theme) — **locked in Phase 57** |
| **Inline Edit** affordance — in-row editable field PATCHing the proposal before approve (D-05) | The inherited workspace patterns (Scaffold / File Table / Lane Card) — **locked in Phase 58/59** |
| **Propose** workspace — light generation view (proposals list + model + confidence + GENERATE ALL, D-01) | Row-body click → rich per-file record slide-in — **Phase 61 (RECORD-01)**; rows are inert-but-present |
| **Bulk "approve all high-confidence"** header action — server-evaluated fixed-threshold predicate (REVIEW-02, D-02/D-03) | Per-stage configurable thresholds + a threshold slider/override UI — **REVIEW-06 (deferred)** |
| **Dedupe** workspace — duplicate-group cards + radio keeper-select + auto-keep bulk (REVIEW-03, D-07) | A unified below-the-line **Audit rail node** — **deferred** (reuse existing `/audit/` page) |
| **Cue** workspace — `.cue` preview cards + Approve, gated on a matched tracklist (REVIEW-04, D-08) | ⌘K / Agents / first-run empty state — **Phase 61** |
| **Applied/reversible row state** + in-workspace **UNDO** control over existing endpoints (REVIEW-05, D-04) | a11y depth / responsive rail-collapse / dead-template removal — **Phase 62** |
| Workspace-level copywriting (CTAs, empty/gated states, confirm copy) | `@alpinejs/focus` — a **Phase-61 dep**; do NOT introduce it here |

**SCOPE NOTE (the one difference from 58/59):** Phase 60 is **NOT** strictly no-backend-change. It is
**explicitly sanctioned to add exactly TWO thin UI-serving endpoints** — the D-02 server-evaluated
bulk-approve-predicate and the D-05 inline-Edit PATCH — both thin routes over unchanged apply/generation
logic. Everything else (dedupe, cue, audit, reversibility) rides existing endpoints with zero backend
change. Do not cut these two; do not expand beyond them. This is a **UI contract**, not a backend spec —
it describes the visual/interaction surface those two endpoints serve.

## Color (inherited base + the diff before/after tints; NO new destructive-action color)

The Phase-57 C3 palette is inherited **verbatim** (`phaze-bg` dominant 60%, `phaze-panel` +
`phaze-border` secondary 30%, `blue-500` accent 10%, `emerald` success, `amber` attention, `rose`
status/failed). Phase 60 adds **no new base tokens and no new accent reservation of blue.** It maps the
existing semantic colors onto the diff and the approval-state signals.

### The diff before→after tints (NEW — the load-bearing Review signal, from prototype `.del`/`.add`)

| Facet | Treatment (dark) | Light variant | Meaning |
|-------|------------------|---------------|---------|
| **BEFORE** (current value, struck) | `bg-rose-500/10 text-rose-300 line-through` | `bg-rose-50 text-rose-600 line-through` | the value being replaced (prototype `.del`) |
| **AFTER** (proposed value, highlighted) | `bg-emerald-500/10 text-emerald-300` (no line-through) | `bg-emerald-50 text-emerald-600` | the proposed value that will be written (prototype `.add`) |
| Arrow between | `text-gray-600` | `text-gray-400` | `→` glyph, `col-2 auto` |

> **Rose-as-status, NOT destructive-action (carried forward from Phase 58/59):** the struck rose BEFORE
> value marks a **removed/old value** — a status tint, exactly as Phase 58/59 render `rose` for a
> failed/removed state. It is deliberately paired with `line-through` so the removal reads without relying
> on hue alone (WCAG 1.4.1). It is **not** a destructive-action button color. This keeps rose consistent
> across the milestone (removed / failed = rose status tint).

### Approval-state colors (per-row + applied lifecycle)

| Signal | Word | Treatment | Wired to |
|--------|------|-----------|----------|
| per-row **APPROVE** (commit, reversible) | "APPROVE" | **`bg-emerald-600/90 hover:bg-emerald-500 text-white`** filled button (prototype `diffRow`) | `proposals.py:168` approve (Rename/Move) · tag-write approve (Tag) |
| **EDIT** (open inline edit) | "EDIT" | inherited **secondary** button — `bg-phaze-panel border-phaze-border hover:bg-white/5 text-gray-300` | toggles inline edit (D-05 PATCH on save) |
| **SKIP** (dismiss without applying) | "SKIP" | inherited secondary — `text-gray-400` | **existing `reject` status** (D discretion: reject is the sensible default) |
| row **approved** (queued to apply) | "approved" | `text-emerald-300` word + faded row (`opacity-70`) + in-row **UNDO** | — |

### Bulk "approve all high-confidence" — the amber-attention treatment (REVIEW-02, high-stakes)

The bulk header action is the phase's highest-stakes control (it applies MANY rows at once). It is made
**visually distinct** from the safe per-row emerald APPROVE by the inherited **amber attention** color —
`amber` is already the Phase-57 **Review & Apply rail-group color** (pending-approval), so this is a
coherent reuse, not a new reservation:

| Element | Treatment |
|---------|-----------|
| Bulk header button | secondary height (`h-9 px-3`), **amber-bordered**: `border-amber-500/40 text-amber-200 hover:bg-amber-500/10` + Jura uppercase `tracking-wider` |
| Confirm affordance | mandatory `hx-confirm` naming the **fixed predicate + live count** (copy below) before firing |
| Busy gate | `:disabled` bound to the relevant `$store.pipeline.*Busy` count (R-4 inherited) so a second bulk apply can't fire mid-churn |

**Destructive-action color:** **none** — consistent with Phase 57/58/59. Every apply in this phase writes
an audit row and is **reversible** (D-04: `ExecutionLog`/`TagWriteLog`/dedupe-resolution + undo/undo-all),
so no action is truly irreversible. The highest-stakes affordances (bulk apply, Move-files physical apply)
are made distinct via the **amber-attention border + mandatory confirm + busy-gate**, NOT a
destructive-red button. Per-row APPROVE stays emerald (safe, single, undoable). This holds the milestone's
"reversible ⇒ no destructive-red" discipline while satisfying "make bulk/high-stakes affordances visually
distinct and confirmed."

**Confidence / discrepancy tier colors** (reused from Phase 59, for the Propose `Conf` cell and the
tag-discrepancy signal): `≥90%` `emerald-300` · `70–89%` `amber-300` · `<70%` `gray-500`; tag "no
discrepancies" `emerald-300` · "has discrepancies" `amber-300`.

## NEW Pattern 1 — Unified before→after Diff Partial (REVIEW-01, D-06)

**ONE** Jinja partial (e.g. `pipeline/partials/_diff_row.html`) reused across THREE queues — Rename/Path,
Tag-write, Move-files — each passing a different facet through the same shape (D-06). This is the single
authoritative Review interaction. Tokenized from the prototype `diffRow()`:

```
<div id="{row_id_prefix}-{id}" class="px-6 py-3 border-b phaze-border hover:bg-white/5 cursor-pointer"
     x-data="{ editing:false, val:'{{ proposed|e }}' }">      {# inert row-body click (Phase 61); Alpine LOCAL edit state #}
  <div class="flex items-center justify-between">
    <div class="text-gray-200 text-sm truncate" title="{{ original_path }}">{{ file }}</div>
    <div class="flex gap-1.5 shrink-0 ml-4" x-show="!editing">   {# ACTION CLUSTER — pending state #}
      <button ...APPROVE (emerald-600 fill)   hx-post=".../approve">APPROVE</button>
      <button ...EDIT (secondary)             @click="editing=true; $nextTick(()=>$refs.edit.focus())">EDIT</button>
      <button ...SKIP (secondary)             hx-post=".../reject" hx-confirm="Skip this proposal?">SKIP</button>
    </div>
    <div class="flex gap-1.5 shrink-0 ml-4" x-show="editing">    {# ACTION CLUSTER — inline-edit state #}
      <button ...SAVE EDIT (emerald-600 fill)  hx-patch=".../{id}" hx-include="closest div" hx-target="#{row_id_prefix}-{id}" hx-swap="outerHTML">SAVE EDIT</button>
      <button ...DISCARD (secondary)       @click="editing=false; val='{{ proposed|e }}'">DISCARD</button>
    </div>
  </div>
  <div class="mt-2 grid grid-cols-[1fr_auto_1fr] gap-3 items-center text-xs">
    <code class="del ...">{{ before }}</code>                    {# rose struck — current value #}
    <span class="text-gray-600">→</span>
    <template x-if="!editing"><code class="add ...">{{ after }}</code></template>   {# emerald — proposed #}
    <template x-if="editing">
      <input x-ref="edit" x-model="val" name="proposed"
             class="add-edit bg-phaze-panel border-phaze-border rounded px-2 py-1 mono text-xs w-full outline-none focus:ring-2 focus:ring-blue-500">
    </template>
  </div>
</div>
```

**Diff-row anatomy contract:** The before→after diff grid is the focal content of every row; the amber bulk-header action is the workspace's focal interactive signal.
- **Top line:** truncating file name (`title=` full path) + the right-aligned **3-action cluster**
  (Approve · Edit · Skip), `shrink-0`.
- **Body:** `grid grid-cols-[1fr_auto_1fr]` — BEFORE `<code>` (rose struck) · `→` · AFTER `<code>`
  (emerald). Both `block truncate` so long paths/tags stay one line.
- **Per-facet mapping (the "three data sources", D-06):**

  | Queue | Rail node | `before` | `after` | Source |
  |-------|-----------|----------|---------|--------|
  | **Rename / Path** | `rename` | current `original_filename` | `proposed_filename` | `RenameProposal` (`proposal.py`) |
  | **Move files** | `move` | current `source_path` | `proposed_path` | `RenameProposal.proposed_path` (`preview.py` → `/s/move`) |
  | **Tag write** | `tagwrite` | current tags (`artist: «empty» · title: …`) | proposed tags (`artist: deadmau5 · …`) | computed comparison (`tags.py` `_build_comparison`) |

- **Rename/Path vs Move split (D-06 discretion — LOCKED):** these are **two separate rail-node
  workspaces** over the one `RenameProposal` source (NOT one workspace with a name/path toggle). The rail
  already carries distinct `rename` and `move` nodes (Phase 57 rail order); superseding one placeholder
  per node keeps the single-poll OOB fanout to one fragment each and keeps the diff partial cleanest.
- **Row is inert-but-present (R-1):** `cursor-pointer` + a stable `{row_id_prefix}-{id}` id + hover, but
  the **row-body click is UNBOUND** (row → record is Phase 61, matching 58/59). Only the in-row Approve/
  Edit/Skip controls are live.
- **DB→render trust boundary:** every `before`/`after`/file cell is autoescaped (never `| safe`).

### Inline Edit affordance (D-05 — LOCKED: in-place field, NOT a modal)

**EDIT expands the row in place** — the AFTER `<code>` is swapped (Alpine `x-if`) for a text `<input>`
pre-filled with the proposed value, and the action cluster swaps Approve/Edit/Skip → **SAVE EDIT / DISCARD**.
This is a **small inline form**, not a modal and not a row-expansion panel — it reuses the row's own
before/after grid cell:

- Alpine **local** row state `x-data="{ editing, val }"` (per-row island); focus moves to the input on
  open via `$nextTick`. No `@alpinejs/focus` (Phase-61 dep) — plain `.focus()`.
- **SAVE EDIT** → `hx-patch` to the D-05 endpoint updating `proposed_filename` (Rename) / `proposed_path`
  (Move) — the Tag queue's Edit is out of the initial cut unless the tag PATCH is trivially symmetric;
  planner decides, but the **UI shape is identical**. Targets **only its own row** (`hx-target` = the row
  id, `hx-swap="outerHTML"`) so the swap never touches sibling rows or the operator's scroll.
- **DISCARD** resets `val` and closes — no request.
- The edited value is validated/persisted server-side (thin PATCH over the existing status-update path,
  D-05); the UI does not re-run the LLM.

## NEW Pattern 3 — Bulk "Approve All High-Confidence" Header Action (REVIEW-02, D-02/D-03)

A **header-level** bulk button on each of the three diff queues (Rename/Path · Tag-write · Move-files),
rendered in the `_workspace_scaffold.html` `actions` slot. It carries the amber-attention treatment
(Color § above) + a mandatory confirm + a busy-gate.

**The critical server-predicate contract (REVIEW-02 — the stale-bulk fix):** the button sends a
**server-evaluated predicate** (action + fixed threshold) — the server **re-queries** pending rows
matching the predicate at submit time and acts on them. It **NEVER** sends a client-built row-id list
(`selectedRows` / `proposal_ids`). This mirrors the `tracklists.py:632` `reject-low` pattern (D-02). The
UI therefore posts a **thresholdless, id-less** request (the predicate is fixed server-side); the button
carries no accumulated selection state — which is exactly why the counts-only poll (R-2) cannot corrupt
it.

| Queue | Bulk button copy (Jura uppercase) | Fixed server predicate | Threshold source |
|-------|-----------------------------------|------------------------|------------------|
| **Rename / Path** | **APPROVE ALL ≥90% CONFIDENCE** | pending `RenameProposal.confidence ≥ 0.9` | D-02 (fixed 0.9) |
| **Move files** | **APPROVE ALL ≥90% CONFIDENCE** | pending `RenameProposal.confidence ≥ 0.9` (Move rides the same `RenameProposal.confidence`) | D-02 |
| **Tag write** | **APPROVE ALL WITH NO DISCREPANCIES** | pending tag-write files whose comparison has `discrepancies == 0` | D-03 (no-discrepancy = high-confidence analog) |

- The button copy **communicates the fixed predicate in words** (`≥90% confidence`, `no discrepancies`)
  — there is **no threshold slider / no override UI** (that is REVIEW-06, deferred). The threshold is not
  operator-editable in v7.0.
- Wired to the **NEW D-02 bulk-approve-predicate endpoint** (thin route over unchanged apply logic).
- **Confirm copy** (mandatory `hx-confirm`) names the predicate + the live count — see Copywriting.

## NEW Pattern 5 — Cue Preview Card (REVIEW-04, D-08)

Generated `.cue` sheets rendered as **preview cards** in `grid grid-cols-2 gap-4 p-6`, each with the

```
</div>

{# GATED — no matched tracklist yet (the ineligible empty state) #}
<div class="rounded-xl bg-phaze-panel border phaze-border p-4 opacity-60">
  <div class="text-gray-300 text-sm mb-2">{{ set_name }}.cue</div>
  <div class="text-xs text-gray-500">awaiting tracklist match…</div>
</div>
```

**Contract:**
- **Eligible card:** the `.cue` preview `<pre>` (mono 11px, `bg-phaze-bg` inset) + an emerald **APPROVE**
  wired over the existing cue-generation path (`cue.py` `_build_cue_tracks` preview; approve via the
  existing generation path, D-08). EDIT is optional (§7 says "Approve/Edit") — if shipped it is the

- Zero backend change (D-08). The prototype's `EXPORT APPROVED` header action maps to a distinct
  export feature with no clear existing endpoint — **omit it**; the Cue workspace ships per-card Approve
  only (a bulk approve is not required by REVIEW-04).

## NEW Pattern 6 — Applied / Reversible Row & Audit Surface (REVIEW-05, D-04)

Reversibility is surfaced **in the workspace**, not as a new rail node (the below-the-line unified Audit
node is deferred; Phase 57 already links the existing `/audit/` page below the line).

- **Row lifecycle (Rename/Move/Tag diff rows):**
  `pending` (diff + Approve/Edit/Skip) → `approved` (queued to apply — emerald "approved" word,
  `opacity-70`, in-row **UNDO**) → `applied` (audit row written — emerald "applied ✓", in-row **UNDO**).
- **In-row UNDO control:** the inherited **neutral secondary** button (never a destructive-red fill),
  wired to the existing reverse paths (`proposals.py:218` undo for rename/move; the dedupe
  `undo`/`undo-all`; tag-write reverse over `TagWriteLog`). Applied-vs-approved state is derived from the
  existing `ExecutionLog` / `TagWriteLog` reads (D-04) — a read concern for the planner, not a new query.
- **"One audit row per apply" (SC#5)** is an assertion on existing writes
  (`ExecutionLog`/`TagWriteLog`/dedupe-resolution), not a UI element — the UI merely surfaces the applied
  state + undo.
- **Unified log:** the existing **`/audit/`** page (`execution.py:350`) is reached via the Phase-57
  below-the-line rail **Audit log** link. Phase 60 does **not** restyle `/audit/` and does **not** add an
  Audit rail node.
