## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator → migration | The `-x reattribute_to=<id>` override is operator-supplied input that selects the reattribution target; it crosses into raw SQL. |
| migration → production data | A destructive, irreversible data migration touching production ownership (files/scan_batches agent_id + a RESTRICT-FK agents delete). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-89-02-01 | Tampering | `-x reattribute_to` override → SQL | mitigate | Parameterize via `sa.text(...).bindparams(...)` — never f-string the id; validate against `agents` (exists + kind='fileserver' + not revoked) BEFORE use. |
| T-89-02-02 | Tampering/Repudiation | wrong-target mass reattribution | mitigate | Auto-detect requires EXACTLY one non-revoked fileserver else abort; `COUNT=0` assert before the sentinel DELETE; single transaction rolls back on any raise (D-01/D-09). |
| T-89-02-03 | Denial of Service / data-loss | partial/aborted migration state | mitigate | Whole body runs in one Alembic transaction (transaction_per_migration); any `raise` (invalid override, 0/>1 fileserver, COUNT≠0) rolls back — the sentinel DELETE is never reached in a bad state. |
| T-89-02-04 | Denial of Service | RESTRICT-FK sentinel delete with rows still referencing it | mitigate | Reattribute-before-delete ordering + the COUNT=0 gate guarantee no `files`/`scan_batches` still reference the sentinel when `DELETE FROM agents` runs. |
| T-89-02-05 | Tampering | live-batch unique-index collision (Pitfall 1) | mitigate | DELETE the legacy `status='live'` watcher batch FIRST (D-03 refined) so the bulk `UPDATE scan_batches` cannot create a second live row for the target under `uq_scan_batches_agent_id_live`. |
| T-89-02-SC | Tampering | package installs | accept | No packages installed (uses existing alembic/sqlalchemy); RESEARCH Package Legitimacy Audit = N/A. |
