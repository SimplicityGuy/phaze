## 1. Transport decision (RESEARCH Q1 / Assumption A2)

### DECISION: **Option A — parent-side Queue-drainer over a `multiprocessing.Manager().Queue()` proxy.**

Plan 04 MUST wire the per-window `progress_cb` for the local + A1 (pebble) lanes as a
picklable Manager-queue proxy passed into the child, drained parent-side, with the
**future-done sentinel** teardown proven below. The k8s-lane `asyncio.to_thread` +
`run_coroutine_threadsafe` bridge is settled separately by RESEARCH Pitfall 3 (not re-spiked
here; no event loop / no pebble pool in that lane).

### Why Option A (one-paragraph justification)

The Queue-drainer keeps a **single HTTP surface** (`agent_client.post_analysis_progress`),
inheriting the existing tenacity retry funnel and `AgentApiError` hierarchy (D-11/D-12/D-16)
for free, and keeps `analyze_file` HTTP/pickle-free (only an `(int, int)` tuple crosses the
pebble boundary). Its one real risk — that a SIGKILLed child could **hang** the parent
drainer and stall a worker slot (Assumption A2, threat **T-57.1-30**) — was the spike's
primary unknown and is now **disproven**: the drainer tears down within a bounded deadline
when the child is killed. A `multiprocessing.Manager().Queue()` proxy (NOT a raw
`multiprocessing.Queue`) is required: the manager process owns the queue, so a SIGKILL of the
child cannot wedge an internal feeder lock; the parent can still `get`/`put` after the child
dies. Option B (child-side sync `httpx.Client`) also proved functional but was **rejected**
because it duplicates the POST surface, couples the pebble child directly to the bearer token,
and forgoes the tenacity retry reuse — a second HTTP code path with no offsetting benefit now
that A2 is settled.

### Teardown mechanism Plan 04 MUST reuse

```
future = pool.schedule(analyze_file, args=[...], kwargs={... "progress_cb via queue ..."}, timeout=...)
afut   = asyncio.wrap_future(future)
afut.add_done_callback(lambda _f: progress_q.put(_SENTINEL))   # fires on success OR SIGKILL→TimeoutError
drainer = asyncio.create_task(_drain_until_sentinel(progress_q, deadline, post_stub=client.post_analysis_progress))
```

- The drainer offloads the blocking `queue.get(block, timeout)` via `asyncio.to_thread` so the
  event loop is **never** blocked.
- The `add_done_callback` runs on the loop the instant the child future completes **or raises**
  (`builtins.TimeoutError` on SIGKILL, `pebble.ProcessExpired` on crash); it pushes a
  string-tagged `_SENTINEL` that can never collide with a real `(analyzed, total)` tuple.
- The drainer breaks on the sentinel → **bounded teardown on both the crash and the clean-finish
  path** (one teardown path for both, proven by two separate tests).
- A hard `deadline_sec` backstop guarantees fail-fast even if the sentinel were somehow dropped.

### D-04 throttle home

The ~5s throttle (`_should_post(last, now, interval)`) lives in the **bridge/drainer**, keyed
on `monotonic()`, with an unconditional final flush — NOT in `analyze_file` (keeps the compute
seam pure). Belt-and-suspenders: the existing completion `put_analysis` already writes the final
`fine_windows_analyzed/total`, so the bar reaches 100% from completion even if the last throttled
progress POST is dropped.

| Test (`-k transport`) | Proves |

| `test_transport_queue_drainer_observes_progress_and_is_kill_safe` | ≥2 `(analyzed,total)` counts drained from a real pebble child; child SIGKILLed by a 3.0s pebble timeout (`TimeoutError`); drainer joins within a bounded 10s wait — **no hang** (A2 / T-57.1-30). |

Residual risk: **LOW.** The synthetic worker reproduces the pebble pickle boundary + SIGKILL

## 2. Crash-mid-run idempotency (PROG-02 / RESEARCH Q3) — PROVEN

Against the **existing, unmodified** `put_analysis` write path on **real Postgres** (no
production change at this wave), the spike proves a file killed mid-analysis re-runs to a state
**byte-identical to an uninterrupted control run**:

| Truth (must_have) | Evidence (`-k idempotent`, GREEN) |
|---|---|
| Kill-safe transport (A2) | §1 above (transport group). |
| Re-run final aggregates + window set identical to an uninterrupted control | `test_idempotent_crash_midrun_rerun_matches_control`: a partial row (NULL bpm, analyzed 3<10, `sampled=True`) + 3 stale `analysis_window` rows are overwritten by the re-run; final aggregate tuple and `(tier, window_index)` window set **equal** a separate clean-run control file. |
| `FileState` → `ANALYZED` exactly once; no dup/orphan windows | Same test: terminal state `ANALYZED`; exactly **1** `analysis` row (file_id UQ); window count == control count; zero duplicate keys; the 3 stale `bpm=99.0` windows are **gone** (delete-then-insert replace, not append). |
| ≥2 distinct mid-flight counter values observable | Proxied by the transport group (≥2 drained counts). Real long-file confirmation = Task 3. |
| Re-enqueue safe under Phase 32 | Same test asserts the **state-only** done-predicate (`FileRecord.state IN {ANALYZED, ANALYSIS_FAILED}`, `tasks/reenqueue.py`) classifies the file as done **only after** the terminal re-run — the leftover partial row never mis-drives recovery. |
| Idempotent under replay | `test_idempotent_repeat_rerun_no_duplicate_windows`: a double identical PUT leaves one analysis row and a single non-duplicated window set. |

Conclusion: **PROG-02 is structurally safe.** D-01's "the counter is just overwritten on re-run"
holds, and the partial row cannot leak into recovery. (The proposal-convergence KEY-RISK leak —
a partial row satisfying `exists(AnalysisResult)` — is a SEPARATE concern handled by Plan 02's
`analysis_completed_at` discriminator + Plan 03's gate fix; it is NOT a `put_analysis`
idempotency property and is out of scope for this spike.)
