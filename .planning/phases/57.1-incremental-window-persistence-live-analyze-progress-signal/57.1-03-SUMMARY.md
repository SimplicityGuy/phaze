# Phase 57.1 Plan 03: Counter-only Progress Endpoint + Client Method Summary

**Built the single lane-agnostic HTTP surface for mid-flight analyze progress: `AnalysisProgressPayload`, a counter-only `post_analysis_progress` sibling handler on the existing `agent_analysis` router that upserts ONLY the two fine-window counts (with zero completion side effects), and a best-effort `agent_client.post_analysis_progress` method that returns None on failure so a dropped POST never fails the analysis job.**

- **Duration:** ~15 min

- **Files modified:** 5 (3 source, 2 test)
- **Tests:** 55 passing across the two touched test files (12 new)

## Accomplishments

- **Counter-only endpoint (D-01/D-02, T-57.1-03):** `POST /api/internal/agent/analysis/{file_id}/progress` upserts ONLY `fine_windows_analyzed`/`fine_windows_total` on the `analysis` row via `pg_insert…on_conflict_do_update` on the `file_id` UQ. A START(0,N) then bump(k,N) produces exactly ONE row with the counter advancing. The handler is a **sibling** of `put_analysis` — it copies the upsert mechanism but writes NO `analysis_window` rows, NEVER flips `FileState`, NEVER clears the scheduling ledger, NEVER deletes the staged S3 object, and NEVER stamps `analysis_completed_at`. Proven by a test asserting state stays `DISCOVERED`, no window rows are added, the ledger row survives, and `analysis_completed_at` stays NULL after a progress POST.
- **Partial row stays out of proposals:** because the endpoint leaves `analysis_completed_at` NULL, Plan 02's convergence gate (`analysis_completed_at IS NOT NULL`) keeps the partial in-flight row out of `generate_proposals` — the KEY RISK safety net from wave 2 holds.
- **Strict, spoof-proof wire schema (T-57.1-01/02):** `AnalysisProgressPayload` uses `extra="forbid"` with REQUIRED `ge=0` fine counts. A body smuggling `agent_id`/`file_id` is a 422; a negative or missing count is a ValidationError. `agent_id` comes from `get_authenticated_agent` (the dep, never the body); missing bearer is 401.
- **Best-effort client method (D-16):** `agent_client.post_analysis_progress` POSTs the counts through `self._request` (inheriting the tenacity retry policy + `AgentApiError` hierarchy — no bespoke retry loop) and returns None. A 4xx surfaces immediately without retry; the caller swallows after retries so a dropped progress POST never fails the job (the completion `put_analysis` writes the final count regardless).

## Decisions Made

- **Fine-only counts, required** (CONTEXT D-01 Discretion): fine-only satisfies WORK-04; both counts are required because a progress POST always carries both. Coarse counts intentionally omitted.
- **Sibling handler, not a reuse** (RESEARCH Pattern 1 / T-57.1-03): the only structurally sound way to guarantee no completion side effect can leak via the progress path is a co-located sibling that lifts the upsert idiom but contains none of `put_analysis`'s completion statements. The docstring documents every stripped side effect; the executable body contains none of them (verified by grep — the only matches are in the docstring).
- **TYPE_CHECKING typing for the client method**: matched `put_analysis`'s convention (payload type in the `TYPE_CHECKING` block, used unquoted in the signature) rather than a quoted forward-ref + lazy runtime import, since the payload is passed as an instance and no runtime import is needed.
