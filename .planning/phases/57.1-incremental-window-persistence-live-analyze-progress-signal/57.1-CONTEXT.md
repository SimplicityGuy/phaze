# Phase 57.1: Incremental window persistence & live analyze progress signal - Context

**Gathered:** 2026-06-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Make audio-analysis progress observable **mid-flight** so the Phase 58 Analyze workspace can show a real per-file in-flight progress bar (WORK-04). Today `analyze_file()` (`src/phaze/services/analysis.py:520`) runs the whole file — both fine and coarse passes — and returns **once** at the end; the result reaches the control-plane DB via a single `put_analysis` PUT at completion (`src/phaze/routers/agent_analysis.py:126`), which replaces the file's `analysis_window` rows atomically. In-flight files therefore have **no** progress signal.

This phase adds an incremental **progress count** (not detail rows) that advances during the run and is readable per-file, across all three execution lanes that call the same `analyze_file()`: the local SAQ worker (`src/phaze/tasks/functions.py:168`), the A1 compute agent, and the k8s one-shot job (`src/phaze/job_runner.py:218`).

**This is the ONE deliberate, scoped backend change in v7.0** (approved 2026-06-29) — isolated here so every v7.0 UI phase (incl. Phase 58, which `depends_on` this) stays presentation-only. Delivers PROG-01..03.

**Scope fence (do NOT cross):** NO new queue/task/routing semantics; NO change to representative aggregates (median BPM, modal key, dominant mood/style), the windowed-analysis algorithm (Phase 31 strided caps), or the final `ANALYZED` `FileState` flip. Window **detail** rows keep their atomic-at-completion write.

**First plan task = a spike** confirming incremental progress-counter persistence + crash-mid-run idempotency on a real long (multi-hour) file before the design is locked into the production write path.
</domain>

<decisions>
## Implementation Decisions

### Mid-flight granularity (PROG-01)
- **D-01: Counter-only mid-flight — window detail rows stay atomic.** During analysis, only a lightweight **count** advances (`fine_windows_analyzed` / `fine_windows_total`; coarse counts optional — see Discretion). The heavy `analysis_window` **detail** rows continue to land atomically at completion via the existing `put_analysis` replace path. This keeps Phase 31/32 idempotency essentially untouched and makes PROG-02 trivial (the counter is just overwritten on re-run; there are no partial detail rows to reconcile). The Phase 58 bar needs only the count, so this fully satisfies the requirement.
- **Reconciliation note for the planner:** PROG-01's ROADMAP/REQUIREMENTS wording says "incremental window **persistence**" — under D-01 the accurate description is incremental **progress-counter** persistence (detail rows are NOT written incrementally). Update PROG-01's wording (and the Phase 57.1 ROADMAP goal's "persist `analysis_window` rows … incrementally" clause) to match D-01 when planning, or add a one-line note; do not implement incremental detail-row writes.

### Transport across local / A1 / k8s lanes (PROG-01)
- **D-02: One new lightweight internal progress-POST endpoint, used by all lanes, fed by an `analyze_file` per-window callback.** Add a progress callback parameter to `analyze_file`'s per-window loop (`_analyze_fine_windows` / `_analyze_coarse_windows`). Each lane reports through ONE new small internal endpoint (e.g. `POST /api/internal/agent/files/{file_id}/analysis-progress`) that upserts the count — reusing the established agent→control incremental-POST pattern (`post_exec_batch_progress`, `heartbeat` in `services/agent_client.py`, Phase 28). The local in-process worker also goes through this HTTP path (it already POSTs `put_analysis` via `agent_client`), so there is **one** code path, lane-agnostic. Do NOT piggyback on heartbeat (wrong semantics; k8s one-shot job isn't a heartbeating agent) and do NOT special-case the local lane with a direct DB write (two paths = divergence risk).

### Counter home + PROG-03 read shape (PROG-03)
- **D-03: Upsert a partial `analysis` row at analysis START; Phase 58 reads the count off that row.** The progress endpoint upserts the file's `analysis` row when analysis begins (`fine_windows_total` set, `fine_windows_analyzed` = 0, bumped mid-flight), reusing the existing `analysis.file_id` UQ natural key and `put_analysis`'s field-level last-write-wins replace semantics. Completion overwrites the **same** row with final aggregates + window rows. Phase 58 reads `fine_windows_analyzed`/`fine_windows_total` off the one analysis row — unified for in-flight and completed files; an idempotent re-run just overwrites it.

### Cadence (PROG-01)
- **D-04: Per-window callback, time-throttled to ~once / 5s, always flush the final count.** Fire the progress callback per window but skip the POST if the previous one was <~5s ago; always send the final count before/at completion. Given Phase 31 caps (≤~60 fine + ≤~30 coarse windows per file regardless of duration) this is naturally cheap; the throttle only avoids a burst on short/fast files. (Exact interval is a tunable default — planner may pick 3–5s.)

### Claude's Discretion
- Whether to also surface a **coarse** progress count (`coarse_windows_analyzed`/`total`) or fine-only for the bar — fine-only is sufficient for WORK-04; coarse is a cheap add if it falls out naturally.
- Exact endpoint name/path + payload schema (follow the `agent_*` internal-endpoint + bearer-auth conventions; `agent_id` from the auth dep, never the body — AUTH-01).
- Throttle interval (3–5s) and whether throttling lives in the callback, the worker, or the client.
- Migration: the `analysis` columns (`fine_windows_analyzed`/`fine_windows_total`) already exist (`src/phaze/models/analysis.py:28-29`) — confirm no new migration is needed for the counter (only possibly for a "started_at"/in-progress marker if one is chosen).

### KEY RISK to engineer around (carry into plan must_haves)
- **A partial `analysis` row (NULL aggregates, analyzed < total) must NOT be treated as a completed analysis** by proposals / search / sort / the `ANALYZED` gate. Those must stay gated on the `FileState` `ANALYZED` flip (which still happens only at completion), not on mere analysis-row existence. Add a regression test asserting an in-progress analysis row does not leak into proposals/search. This is the one place D-03 could regress existing behavior.
</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **`analyze_file` per-window loop** — the single seam for a progress callback; one change covers all 3 lanes because they all call this function.
- **`agent_client` POST patterns** (`put_analysis`/`post_exec_batch_progress`/`heartbeat`) — the new progress POST mirrors these (bearer auth, internal `/api/internal/agent/*` path, `agent_id` from auth dep).
- **`put_analysis` upsert** (`file_id` UQ, field-level `exclude_unset` last-write-wins, window-replace) — D-03 reuses this exact mechanism for the partial-row upsert + final overwrite.
- **Existing `fine_windows_analyzed`/`fine_windows_total` columns** — the counter has a home; likely no migration.

### Established Patterns
- **Agent→control incremental progress POSTs** already exist (Phase 28 exec-batch progress) — this phase is a new instance of a known pattern, not a new architecture.
- **Three execution lanes, one `analyze_file`** — keep the change in the shared function + a shared client method so local/A1/k8s stay uniform.

### Integration Points
- New internal endpoint registered alongside `agent_analysis.py`'s router.
- The progress callback is threaded from `analyze_file` → its callers (`functions.py`, `job_runner.py`) → `agent_client` progress POST.
- Phase 58 (downstream) queries the analysis row's count fields; no Phase-58-side backend work.
</code_context>

<specifics>
## Specific Ideas

- **Counter, not rows** — the entire risk reduction of this phase hinges on D-01: do not write `analysis_window` detail rows incrementally.
- **Spike first** — validate incremental counter persistence + crash-mid-run idempotency (kill the process mid-analysis, re-enqueue, assert clean re-run with no duplicate/partial corruption and the final row identical to an uninterrupted run) on a real multi-hour file before wiring the production path.
- Keep the new endpoint and callback minimal — a count and the file id; resist adding per-window detail to the wire.
</specifics>

None arose as scope creep — discussion stayed within the progress-signal boundary.
