# Phase 57.1 Plan 04: Wire the per-window progress signal through the shared seam + both non-A1 lanes Summary

**Wired the live analyze progress signal end-to-end: `analyze_file` now fires a sync `progress_cb` (START + per-window bump, `len(natural)` denominator) that stays HTTP/pickle-free, the pebble (local + A1) lane bridges it via a picklable `multiprocessing.Manager().Queue()` sink → a kill-safe parent-side drainer → `post_analysis_progress` (throttled, best-effort), and the k8s one-shot lane runs `analyze_file` in `asyncio.to_thread` with a `run_coroutine_threadsafe` fire-and-forget cb — all throttled by a new `analysis_progress_interval_sec` knob, with a dropped progress POST never failing the job and never altering the EXIT_ANALYSIS/EXIT_CALLBACK contract.**

## Accomplishments

- **The shared seam (Task 1, PROG-01):** `_analyze_fine_windows(..., progress_cb=None)` fires `progress_cb(0, len(natural))` BEFORE the loop and `progress_cb(len(fine_windows), len(natural))` after every successful append; `analyze_file` threads it through (keyword-only). The denominator is `len(natural)` — IDENTICAL to the `fine_windows_total` the completion PUT reports (denominator invariant), so the in-flight bar and final coverage agree. `progress_cb=None` is byte-identical to before. `analysis.py` imports no `httpx`/`agent_client` (grep-asserted; `Callable` lives in `TYPE_CHECKING`), so the pebble pickle boundary and the `tests/test_task_split.py` essentia import boundary stay intact.
- **The throttle knob (Task 1, D-04):** `AgentSettings.analysis_progress_interval_sec` (default `5.0`, `ge=0.0`) — the parent/loop-side throttle home; `0` disables throttling, and the final count is always flushed regardless.
- **The pebble lane bridge (Task 2, PROG-01/02):** per 57.1-01 Option A, a module-level picklable `_QueueProgressSink` pushes only an `(analyzed, total)` int-pair onto a `Manager().Queue()` proxy; `_run_analysis_with_progress` launches the pool call as a task, attaches a future-done callback that pushes a string sentinel (on success OR SIGKILL→`TimeoutError`), and runs a parent-side `_drain_progress` that reads counts off-loop via `asyncio.to_thread(queue.get, ...)`, throttles on `monotonic()`, posts best-effort via `ctx["api_client"].post_analysis_progress`, and final-flushes. Teardown is bounded (`wait_for` + cancel backstop) and the Manager is always shut down. `TimeoutError`/`ProcessExpired` re-raise UNCHANGED so the existing terminal `report_analysis_failed` mappings are byte-identical.
- **The k8s lane bridge (Task 3, PROG-01):** `job_runner.run` captures the loop, then runs the blocking `analyze_file` via `asyncio.to_thread(..., progress_cb=cb)` so the loop stays free; `_make_progress_cb` throttles on `monotonic()` and schedules `_safe_post_progress` via `asyncio.run_coroutine_threadsafe(..., loop)` (fire-and-forget — never `.result()`). The START count and the final count (`analyzed >= total`) always post (≥2 mid-flight POSTs). Any cb/POST error is swallowed; `EXIT_ANALYSIS`/`EXIT_CALLBACK`/success exit codes are unchanged.
- **Best-effort everywhere (D-16, T-57.1-23):** both bridges swallow the `AgentApiError` hierarchy (and anything else) so a dropped progress POST never fails the analysis job — the completion `put_analysis` writes the final count, so the bar reaches 100% from completion regardless.

## Decisions Made

- **Fine-only progress** (CONTEXT D-01 Discretion): only the FINE pass is instrumented; coarse is intentionally left out — fine satisfies WORK-04 and avoids a second-denominator UI concern (Pitfall 5).
- **Throttle home = the bridge, not the seam**: `analyze_file` emits unthrottled; the lane drainer/cb collapses bursts on `monotonic()` to `analysis_progress_interval_sec` and always flushes the final count (D-04). The first emission always posts (initial `last_post=0`), giving the bar an early total.
- **Reuse the spike's exact teardown**: the future-done sentinel + bounded deadline backstop from 57.1-01 are reproduced verbatim so a SIGKILLed child never wedges a worker slot (T-57.1-22).
