<objective>
Close the KEY RISK before any partial analysis row can ever be written. D-03 upserts an `analysis` row at analysis START — but `get_proposal_pending_batches` (`services/pipeline.py:1118-1126`) currently treats bare `exists(AnalysisResult)` as "analysis complete". A `METADATA_EXTRACTED` file with a partial START row would satisfy BOTH gate clauses and leak into `generate_proposals` with NULL aggregates.

This plan adds a **completion discriminator** — a new nullable `analysis_completed_at` column (migration 028, RESEARCH Q4 recommended over the zero-migration FileState tighten because it is intention-revealing and gives Phase 58 a crisp in-flight flag), stamped ONLY in the existing `put_analysis` completion branch — and tightens the convergence gate to require it. It lands FIRST (wave 2, before the progress endpoint in plan 03) so the safety net precedes the first partial-row write.

Purpose: Make "row exists ⇒ analysis complete" false safely — the one place D-03 regresses existing behavior.
Output: migration 028 + model column + completion stamp + gate predicate + the mandatory KEY-RISK regression test.
</objective>
