> Visual and interaction contract for the two **Identify** stage workspaces that live inside the
> v7.0 shell: **Track-ID** and **Tracklist**. Generated by gsd-ui-researcher.
>

> verbatim** (which itself inherits Phase 57). Spacing, typography, color tokens, the shell chrome,
> the DAG rail, the header status strip, the ⌘K modal, the OOB single-poll fanout, `$store.pipeline`,

> `docs/superpowers/specs/2026-06-28-ui-redesign-assets/prototype.html` (the `trackid()` /
> `tracklist()` screen functions). **Re-scope override (IDENT-01, carried forward):** the prototype's

## Scope Boundary (read first)

This spec covers the **content of two stage workspaces** that swap into the Phase-57 `#stage-workspace`
target, superseding the `trackid` / `tracklist` `_STAGE_PLACEHOLDER` entries in
`src/phaze/routers/shell.py:69-70` (supersede-in-place; legacy templates stay until CUT-02/Phase 62).
It does NOT re-specify the shell chrome or the four inherited workspace patterns.

| In scope (contract below) | Out of scope (deferred — note only) |
|---------------------------|--------------------------------------|
| **Track-ID** workspace — ONE combined per-file table surfacing existing fingerprint + tracklist signals (IDENT-01) | The shell chrome (header, rail, ⌘K, theme) — **locked in Phase 57** |
| **Tracklist** workspace — 3 step cards (Search · Scrape · Match) + per-set table (IDENT-02) | The four NEW workspace patterns (Scaffold / File Table / Lane Card / Per-file badge) — **locked in Phase 58** |
| Per-engine fingerprint **status** badges (audfprint · Panako: done / failed / pending) — NO invented score (D-01/D-02) | AcoustID acoustic lookup + MusicBrainz recording resolution + numeric fingerprint **score** — **IDENT-03 (deferred)** |
| Tracklist **match state** + **confidence** cell (linked-or-best, D-04) | Row-click → rich per-file record / right detail pane — **Phase 61 (RECORD-01)** |
| Tracklist **step-card** pattern (lane-card visual + per-step ALL trigger, D-05/D-06) | A single "run chain" Search→Scrape→Match orchestrator trigger — **cut (no backend endpoint)** |
| Per-set **track-level coverage** signal (N/M tracks confident, D-07/D-08) | Unified Review & Apply diff/approve gate — **Phase 60** |
| Workspace-level copywriting (CTAs, empty states, status words, confirm copy) | a11y depth / responsive rail-collapse / dead-template removal — **Phase 62** |

Both workspaces are **IA/presentation only — no backend behavior change** (the v7.0 milestone
constraint, restated as a hard boundary in CONTEXT.md). Every endpoint, query, model, and task stage
is reused verbatim. The right detail pane stays the Phase-57 empty placeholder; row→pane wiring and
the rich record are **Phase 61** (Interaction Contract R-1).

## Color (inherited — unchanged from Phase 57/58)

The Phase-57 C3 palette is inherited **verbatim** (`phaze-bg` dominant 60%, `phaze-panel` +
`phaze-border` secondary 30%, `blue-500` accent 10%, `emerald` success, `amber` attention). Phase 59
adds **no new base tokens and no new accent reservations.** It maps the existing semantic colors onto
the two Identify signal families.

### Track-ID — engine status colors (D-01)

Each file row shows two per-engine fingerprint **status** badges (audfprint · Panako), each reading
`FingerprintResult.status`. Status is a **word + color**, never hue-only (WCAG 1.4.1):

| Engine status | Word | Color | Source |
|---------------|------|-------|--------|
| done (`completed` / `success`) | "done" | `emerald-600 dark:emerald-300` | `fingerprint_results.status` |
| failed | "failed" | `rose-600 dark:rose-400` | `fingerprint_results.status` (+ optional `error_message` tooltip — Claude's discretion, D-08 note) |
| pending (no row for `(file_id, engine)`) | "pending" | `gray-500 dark:gray-400` | absence of a `FingerprintResult` row |

> **Rose-as-status (not destructive) — carried forward from Phase 58:** `rose` text here marks a
> **failed engine state**, exactly as the Phase-58 Analyze table renders `analysis_failed` →
> `text-rose-600 dark:text-rose-400`. This is a **status color, not a destructive-action color** —
> Phase 59 introduces no irreversible/delete action (see Copywriting › Destructive). The Phase-57
> "no destructive color in chrome" rule is about *actions*; a failed-state word is a status signal.

### Track-ID — tracklist match state + confidence (D-04)

| Signal | Treatment |
|--------|-----------|
| Tracklist **match state** | word span: "matched" (`emerald-300`) when a tracklist is linked; "candidate" (`amber-300`) when only a best-candidate exists; "no match" (`gray-500`) when none |
| Tracklist **confidence** | `font-mono` `{n}%` (rounded `Tracklist.match_confidence` int): **≥90** `emerald-300` · **70–89** `amber-300` · **<70** `gray-500`; none → "—" `gray-500`. (Linked tracklist's `match_confidence`, fallback to the highest candidate `match_confidence` — `list_tracklists` already orders `desc nulls_last`.) |

### Tracklist — step-card + per-set colors (D-05/D-07)

| Element | Treatment |
|---------|-----------|
| Step-card count numeral | `font-mono font-medium` — `gray-300` neutral; the live **busy** indicator binds to the existing `searchBusy` / `scrapeBusy` / `matchBusy` store keys and shows `blue-300` "running…" when `> 0` |
| Per-set tracklist match state | word span: "found"/"matched" `emerald-300` · "searching…" `blue-300` · "—" `gray-500` (mirrors the prototype `tracklist()` cells, re-tokened) |
| Per-set **track coverage** (N/M confident, D-07) | `font-mono` `{n}/{m}`: full (`n==m`) `emerald-300` · partial `blue-300` · none/`—` `gray-500` |

**Accent reserved-for:** unchanged from Phase 57/58 — Phase 59 makes **no new reservation of blue**.
Blue here is the inherited "in-flight / running" semantic only (per-set "searching…", step busy,
partial coverage), consistent with the Phase-58 in-flight usage. No lane-identity colors apply
(those are Analyze-only).

**Destructive color:** none — Phase 59 introduces no irreversible/delete action (see Copywriting).

table partial — its `columns` / `rows` (cell dicts) / `empty_heading` / `empty_body` /
`row_id_prefix` contract). Composed inside `_workspace_scaffold.html` (`{% call ws.workspace(...) %}`).

**Columns (re-scoped from the prototype `trackid()` — AcoustID/MB dropped):**

| # | Column | Cell content |
|---|--------|--------------|
| 1 | **File** | `original_filename`, `mono`, `title=original_path` (the truncating first cell) |
| 2 | **audfprint** | status word (done/failed/pending — Track-ID engine colors above) |
| 3 | **Panako** | status word (done/failed/pending) |
| 4 | **Tracklist** | match-state word (matched / candidate / no match) |

  (below), not here. The prototype's `IDENTIFY PENDING` button maps to the dropped AcoustID flow and
  is therefore **cut** (no backend endpoint exists; adding one is IDENT-03). The scaffold's `actions`

## NEW Pattern B — Tracklist Step Card (D-05/D-06, IDENT-02)

THREE sequential step cards — **Search · Scrape · Match** — in a `grid grid-cols-3 gap-4 p-6`,
**following the Phase-58 lane-card visual pattern** (`rounded-xl border bg phaze-panel p-4`, title
row + count numeral + sub-label), **NOT a horizontal stepper** (a new component, not in the design
system — D-05). Each card extends the lane-card shell with one addition the lane card lacks: a
**per-step ALL trigger button** (D-06).

```
<div rounded-xl border phaze-border bg phaze-panel p-4>
  <div flex justify-between>
    <span text-gray-200>{N · 🔎 SEARCH}</span>                 {# step number + emoji + Jura label #}
    <span mono font-medium text-gray-300>{done / total}</span> {# server-rendered pending/done count #}
  </div>
  <div text-xs text-gray-500 mt-2>{SUB-LABEL}                  {# "{pending} pending" #}
    <span x-show="$store.pipeline.{step}Busy > 0" text-blue-300>· running…</span>  {# live busy, single poll #}
  </div>
  <div mt-3>{ALL TRIGGER BUTTON}</div>                         {# D-06 — secondary button, R-4 guarded #}
</div>
```

| Step | Card title | Count source (server-rendered) | Busy store key (live) | ALL trigger → endpoint |
|------|-----------|--------------------------------|------------------------|------------------------|
| 1 | **🔎 SEARCH** | tracklist search done/pending | `searchBusy` | **SEARCH ALL** → `POST /pipeline/search-tracklists` |
| 2 | **📄 SCRAPE** | `get_scrape_pending_tracklists` count | `scrapeBusy` | **SCRAPE ALL** → existing scrape bulk endpoint |
| 3 | **🔗 MATCH** | `get_match_pending_tracklists` count | `matchBusy` | **MATCH ALL** → existing match bulk endpoint |

**Contract:**
- Step counts are **server-rendered at fragment render time** (like the Phase-58 k8s lane
  `capacity_value`), sourced from the existing `get_*_pending_*` helpers + the `scan_search`/`scrape`/
  `match` stage map (`pipeline.py:89-91`). **No new store key is required.** The live **busy** pill
  binds via `x-show`/`x-text` to the existing `searchBusy` / `scrapeBusy` / `matchBusy` store keys
  already seeded in `_workspace_poll_seeds.html` and refreshed by the single `/pipeline/stats` poll.
- "Triggerable from one surface" (IDENT-02) = all three ALL triggers **co-located** on the one
  Tracklist workspace. There is **NO** single "run chain" button — no backend endpoint runs all three
  and adding one would break the no-backend-change rule (D-06; Phase-58 ALL-only precedent).
- Each ALL trigger is wired **verbatim** to its existing per-step endpoint, returns
  `pipeline/partials/trigger_tracklist_response.html` (or the existing trigger_response) into a local
  `*-trigger-response` sink, and carries the **R-4 bulk-enqueue guard** (`hx-confirm` + `:disabled`
  busy-gate on its `*Busy` store key).
- Each card is the Phase-58 **secondary** button style (NOT the blue primary, reserved for the rail
  "+ Scan").

> **Do NOT build the prototype's `1 · SEARCH ✓ → 2 · SCRAPE ✓ → 3 · MATCH ⏳` breadcrumb chip row.**
> D-05 explicitly chose the three step **cards** over the horizontal stepper. Sequence is conveyed by
> left-to-right card order + each card's own count/busy state; do not introduce the stepper component.

## NEW Pattern C — Tracklist Per-Set Table (D-07/D-08, IDENT-02)

A table of sets/files **below** the three step cards (D-08 — aggregate cards on top, per-set detail
below, paralleling Phase-58 Analyze: lane cards + file table). **Reuses `_file_table.html` verbatim.**

**Columns (re-tokened from the prototype `tracklist()` table):**

| # | Column | Cell content |
|---|--------|--------------|
| 1 | **Set** | the set/file name, `mono`, `title=path` (truncating first cell) |
| 2 | **Tracklist** | match-state word ("found"/"matched" emerald · "searching…" blue · "—" gray) |
| 3 | **Tracks** | `mono` track-level coverage **N/M confident** (D-07), colored by full/partial/none |
| 4 | **Matched to file** | linked/auto-linked state word ("matched" emerald · "—" gray) |

**Contract:**
- **Per-set progress = track-level coverage** (D-07): **N/M tracks confident** within the set's linked
  tracklist, derived from `TracklistTrack.confidence` over the tracklist's tracks. This is the
  substantive "how well is this set identified?" signal; the chain-step position is already conveyed
  by the aggregate step cards (D-05), so it is **not** duplicated per-row.
- Rows are **inert-but-present** (D-06 / R-1): `cursor-pointer` + stable `row_id_prefix="tracklist-row"`
  + hover; click UNBOUND in Phase 59.
- Read-only over the existing `Tracklist` / `TracklistTrack` reads (`list_tracklists`,
  `_get_tracklist_stats`, `get_tracks`). No new query path.

Per-workspace, operator-facing, terse — Jura uppercase eyebrows/labels + lowercase-sentence body (the
C3 voice). Live numerals come from `$store.pipeline`; the strings below are the static frame.

### Primary actions (stage-action / step-card buttons)

| Workspace | Buttons (Jura uppercase) |
|-----------|--------------------------|
| Track-ID | **none** — read-only consolidated identity view (the prototype's `IDENTIFY PENDING` is the dropped AcoustID flow, IDENT-03/deferred; no trigger this phase) |
| Tracklist | **SEARCH ALL** (Search card) · **SCRAPE ALL** (Scrape card) · **MATCH ALL** (Match card) |

> **Two-word CTA decision (SEARCH ALL / SCRAPE ALL / MATCH ALL):** each keeps its `ALL` object —
> following the Phase-58 locked precedent that bulk-vs-subset scope must be explicit in the label
> (`EXTRACT ALL`, `FINGERPRINT ALL`). The verb names the step (and matches its card title); `ALL`
> names the scope (the whole pending set, since per-file subset enqueue would need a new backend
