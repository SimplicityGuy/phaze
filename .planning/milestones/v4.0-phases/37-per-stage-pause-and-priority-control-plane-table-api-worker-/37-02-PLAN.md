<objective>
Build the runtime control surface: the `apply_stage_control` before-enqueue hook that stamps new jobs with their stage's live priority (and parks them when paused), the raw-`saq_jobs` UPDATE service helpers that mutate the existing backlog, and the factory wiring that activates the hook across all four queue construction sites.

Purpose: This is where REQ-37-1 (pause-on-enqueue park), REQ-37-2 (priority stamping + backlog reorder), and REQ-37-3 (sentinel-guarded resume) become executable. The hook is import-boundary-critical (reads the control table via the queue's psycopg3 pool, NOT SQLAlchemy) so the agent worker — which registers the same hook but never enqueues stage jobs — stays Postgres-layer-clean.

Output: `apply_stage_control` + TTL-cached `_read_stage_control` in `stage_control.py`; `set_stage_priority`/`pause_stage`/`resume_stage` in `services/stage_control.py`; the third `register_before_enqueue` in `build_pipeline_queue`; hook unit tests; an extended import-boundary subprocess test.

Open-Q defaults adopted here (no discuss-phase was run — RESEARCH recommendations): control-state cache TTL = 5 seconds (tunable); pause persists across reboots and re-applies to Phase-32 re-enqueued jobs (the hook re-parks them because it reads live control state); resume un-parks only (it does NOT restore priority — priority is independently stamped/adjusted).
</objective>

Phase-36 factory (src/phaze/tasks/_shared/queue_factory.py, from 36-01):
- `build_pipeline_queue(name: str, url: str, *, cache_redis_url: str, min_size: int = 1, max_size: int = 4) -> PostgresQueue`
- Body registers, in order: `q.register_before_enqueue(apply_project_job_defaults)` then `q.register_before_enqueue(apply_deterministic_key)`, then attaches `q.cache_redis`. This plan appends a THIRD: `q.register_before_enqueue(apply_stage_control)` AFTER `apply_deterministic_key` (so `job.key`/`job.function` are final).
- `PostgresQueue` exposes an async `.pool` (psycopg3 `AsyncConnectionPool`), open by enqueue time. Read: `async with job.queue.pool.connection() as conn: await conn.execute(...)`.

Hook analog (src/phaze/tasks/_shared/deterministic_key.py:86-114): `apply_deterministic_key(job)` — early-return on non-registered function, then best-effort try/except-log-never-raise around the queue-side read (`getattr(job.queue,"redis",None)`). The 37 hook mirrors this exactly but reads the control table via `job.queue.pool` instead of Redis.

Raw UPDATE contract (37-RESEARCH §Code Examples, verified vs saq/queue/postgres.py):
- priority: `UPDATE saq_jobs SET priority = :p WHERE status = 'queued' AND key LIKE :pfx`
- pause:    `UPDATE saq_jobs SET scheduled = :s WHERE status = 'queued' AND key LIKE :pfx`  (s = SENTINEL)
- resume:   `UPDATE saq_jobs SET scheduled = 0 WHERE status = 'queued' AND key LIKE :pfx AND scheduled = :s`  (s = SENTINEL)
- `pfx = f"{STAGE_TO_FUNCTION[stage]}:%"` — build ONLY after validating `stage in STAGE_TO_FUNCTION` (allowlist; threat T-37-01). Never scope by `queue` (stages span all per-agent queues; the global `key` PK disambiguates).
- `status='queued'` guard is what makes drain (active jobs untouched) + no-double-pickup safe (37-RESEARCH §Concurrency Safety).

Hook unit-test analog (tests/test_deterministic_key.py:92-135): build a real `saq.job.Job(function=..., kwargs=...)`, attach a fake queue via `SimpleNamespace`, await the hook, assert mutations. `_BoomRedis`-style fake for the best-effort failure case. `tests/_queue_fakes.py` provides `FakeRedis`.

Import-boundary test analog (tests/test_task_split.py:198-236, 239-266): `test_shared_bootstrap_stays_postgres_free` / `test_model_bootstrap_stays_postgres_free` — subprocess sets `PHAZE_ROLE=agent`, imports the module, asserts `forbidden = ("phaze.database","phaze.tasks.session","sqlalchemy.ext.asyncio")` not in `sys.modules`. Add a parallel case for `phaze.tasks._shared.stage_control`.

Async session execute idiom (routers/pipeline.py:248): `result = await session.execute(stmt)`. The service helpers take an `AsyncSession` and may use `sqlalchemy` (`text`, `AsyncSession`) because they run ONLY in the API/controller process — NOT the agent (unlike the hook).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator `stage` value → raw SQL | `stage` selects the `key LIKE` prefix in the service helpers — injection surface if interpolated |
| enqueue hook → agent process | the hook runs inside the agent worker's queue construction; a DB-layer import here breaks the agent boundary |
| admin UPDATE ↔ worker dequeue | concurrent access to the same `saq_jobs` row |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-37-01 | Tampering (SQL injection) | `services/stage_control.py` UPDATEs | mitigate | Validate `stage` against `STAGE_TO_FUNCTION` allowlist before building `pfx`; bind ALL params via `text(...)` (`:p`/`:s`/`:pfx`); SQL kept as static module constants — no user input interpolated |
| T-37-03 | Tampering (mid-dequeue mutation / double-run) | `saq_jobs` UPDATE vs dequeue | mitigate | `status='queued'` guard on every admin UPDATE + Postgres row lock + dequeue `FOR UPDATE SKIP LOCKED` — a being-picked-up job is unmutatable; no app lock added (proven in Plan 03) |
| T-37-04 | Elevation / boundary break | `apply_stage_control` hook | mitigate | Hook reads control table via `job.queue.pool` (psycopg3) only — no `phaze.database`/`sqlalchemy.ext.asyncio`; enforced by the new `test_stage_control_stays_postgres_free` subprocess test |
| T-37-02 | Denial of Service (stall) | hook best-effort read | mitigate | A control-read failure logs + enqueues unpaused/default rather than blocking the enqueue (safe failure mode); priority never set below 0 (clamp is the endpoint's job, DB CHECK backstops) |
</threat_model>
