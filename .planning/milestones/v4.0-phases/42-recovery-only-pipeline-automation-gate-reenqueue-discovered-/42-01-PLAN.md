<objective>
Build the backend recovery engine: a single idempotent `recover_orphaned_work(ctx)` producer that reconciles ALL eight pipeline stages after a detected queue-loss, plus the shared pending-set service helpers that guarantee recovery and the Phase 39-41 manual DAG triggers read ONE definition of "pending" (anti-drift, D-03).

This is a RUNTIME automation-behavior change. The reframe (42-RESEARCH §Q2): after Phase 36 the SAQ broker is Postgres (`saq_jobs`), so queued/active jobs survive a controller restart — the old "Redis is empty after reboot" premise is obsolete. A genuine queue-loss is now the detectable asymmetry "`saq_jobs` has zero queued+active rows while the domain DB still shows pending work."

Purpose: produce the gated, all-stages, idempotent recovery producer that both the startup hook (Plan 02) and the manual Recover button (Plan 02) will call — identical paths, no drift.
Output: shared pending-set helpers + `count_inflight_jobs` detector in `services/pipeline.py`; `recover_orphaned_work` in `tasks/reenqueue.py`; full unit + integration tests.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| controller worker → saq_jobs | recovery reads the broker table to decide "loss"; reads must be static SQL |
| recovery producer → SAQ queues | a privileged bulk-enqueue surface; doubling is the principal risk |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-42-01 | Tampering | recover_orphaned_work bulk re-enqueue | mitigate | Every enqueue flows through the deterministic-key chokepoint; an overlapping live item dedups to a skipped no-op (Phase-32 doubling class). generate_proposals batch membership is sorted (Task 1) so its set-hash key aligns and dedups. |
| T-42-03 | Information disclosure / injection | count_inflight_jobs + saq_jobs reads | mitigate | Static SQL with literal `status IN ('queued','active')` only; no f-string/operator interpolation (mirrors `_STAGE_BUSY_SQL`). |
| T-42-04 | Denial of service | recovery raising into controller boot | mitigate | count_inflight_jobs degrades to 0 in a SAVEPOINT; NoActiveAgentError skips agent stages; the function never raises (Plan 02 also wraps the startup call). |
| T-42-05 | Tampering (false positive) | conservative detector | accept | A false "loss" that re-enqueues every pending set is backstopped by deterministic-key dedup — live items collapse to no-ops, so a false positive cannot double the queue. Documented in code. |
| T-42-SC | Tampering | npm/pip/cargo installs | accept | This plan installs NO packages (42-RESEARCH: Package Legitimacy Audit SKIPPED). No supply-chain surface added. |
