<objective>
Close gap CR-02 from `25-VERIFICATION.md`: PATCH `/api/internal/agent/execution-log/{id}` against a row already in a terminal state (`COMPLETED` or `FAILED`) currently returns 409 `"execution-log status is terminal"` even when the new status equals the current status. This breaks the canonical idempotent-retry case (agent writes COMPLETED → network drops the response → SAQ retries → agent re-sends same PATCH → server 409s).

The handler's own docstring (`agent_execution.py:99-100`) promises "same-status PATCH allowed for idempotent retry; comparator is strict `<`, NOT `<=`" — but the terminal-state guard at line 111 runs BEFORE the same-status check, so the docstring's contract is not honored when the row is in COMPLETED or FAILED.

End-to-end reproduction (verified in 25-VERIFICATION.md "Behavioral Spot-Checks"):
- POST ExecutionLog with `status="completed"` → row created
- PATCH same id with `{"status": "completed"}` → server returns `409 {"detail": "execution-log status is terminal"}`. **Wrong.**

The fix is one operator: change `if cur in _TERMINAL:` to `if cur in _TERMINAL and new != cur:`. This preserves the D-15 monotonic guard (any attempt to mutate a terminal row's status to a *different* value still 409s) while permitting the canonical idempotent-retry case.

Purpose: Restore the idempotent-retry contract that Success Criterion #3 was written to enforce. Phase 26 (agent-side HTTP client) is blocked on this — shipping a client against a server that 409s legitimate same-status retries would require client-side classification of "is this 409 my own previous write, or a real conflict?" which is exactly the contract the server should be providing.
Output: 1 modified router file (`agent_execution.py`), 1 modified test file (`test_agent_execution.py` gets one new regression test for the canonical case and one supplementary test for the FAILED→FAILED case).
</objective>
