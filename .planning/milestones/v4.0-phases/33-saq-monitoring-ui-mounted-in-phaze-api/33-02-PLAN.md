<objective>
Wire the `/saq` dashboard into the existing `phaze-api` app by mounting `build_saq_app`
INSIDE the lifespan startup — after the controller queue + task_router + redis are wired
and after enumerating non-revoked agents from Postgres — gated by `settings.enable_saq_ui`.
This reuses the exact lifespan-created Queue instances (controller + per-agent), so the
dashboard reads through the same Redis pools with NO second connection pool, and the mount
is in place before the server serves any HTTP request (RESEARCH-VERIFIED: mount-in-lifespan
is served because Starlette wraps its router by reference).

Purpose: deliver the operator-facing deep queue view (`controller` + `phaze-agent-<id>`,
where the bulk `process_file` jobs live) without a new port, service, or dependency.
Output: the lifespan mount block + integration tests proving served/reuse/health-intact/flag.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → reverse proxy → phaze-api `/saq` | The reverse proxy terminates TLS and enforces internal-realm auth on a private LAN. The app receives already-authenticated internal traffic; `/saq` has NO app-layer auth by design. |
| `/saq` sub-app → Redis | The dashboard reads (and can mutate) job state through the lifespan-created `Queue` instances' Redis pools. No new pool, no new bound port. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-33-03 | Tampering / Elevation of privilege | `/saq/api/queues/{queue}/jobs/{job}/retry` and `.../abort` (POST mutation endpoints in saq_web, `saq/web/starlette.py:84-101`) | accept | LOCKED: intentionally unauthenticated at the app layer — the reverse proxy's internal-realm auth on the private LAN is the SOLE access control. The PR description MUST state explicitly that `/saq` includes job retry/abort mutation endpoints and is deliberately app-unauthenticated. If the proxy boundary ever weakens, this becomes the primary risk. The `enable_saq_ui=False` flag is the operator kill-switch. |
| T-33-04 | Information disclosure | Job kwargs/results rendered in the dashboard (`saq/web/common.py:31-37` repr) | accept | Single-user internal homelab tool behind the proxy; no secret-bearing payloads identified in `process_file`/control-plane jobs. Acceptable per deployment model. |
| T-33-05 | Spoofing / boundary bypass | Mount exposed on a new port or bypassing the proxy | mitigate | Mounted in-process at `/saq` on the existing 8000 listener — no new bound port, no standalone `saq --web` (LOCKED). Reverse proxy remains the only external entry. |
| T-33-SC | Tampering | package installs | mitigate | No new package installed; `saq[redis]>=0.26.4` already audited (RESEARCH Package Legitimacy Audit). `saq[web]` deliberately NOT added. |
