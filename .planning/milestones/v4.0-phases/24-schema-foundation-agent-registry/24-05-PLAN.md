<objective>
Update the ingestion service to remain functional after Plan 04's migration 013 swaps `files` uniqueness to the composite `(agent_id, original_path)`. The current `bulk_upsert_files` uses `on_conflict_do_update(index_elements=["original_path"])` which will fail with `there is no unique or exclusion constraint matching the ON CONFLICT specification` once 013 drops `uq_files_original_path` (RESEARCH Pitfall 1). This plan stamps every newly-discovered FileRecord and ScanBatch with `agent_id = "legacy-application-server"` as a Phase 24 placeholder until Phase 25 wires real per-agent attribution through the HTTP API.

Purpose: Preserves the dev-loop continuity between Phase 24 and Phase 25 — `just scan` and any direct callers of `run_scan` keep working. The change is cheap (three edit sites: one constant + one record-dict + one ScanBatch + one on_conflict target) and avoids a stale-tests-pass-but-runtime-fails class of bug. Phase 25 will rewrite the call sites via the HTTP API and remove the `LEGACY_AGENT_ID` placeholder.

Output: Modified `src/phaze/services/ingestion.py` (add module constant, add `agent_id` to discovered records, add `agent_id` to ScanBatch construction in run_scan, update conflict target); modified `tests/test_services/test_ingestion.py` (existing tests updated + one new `test_bulk_upsert_same_path_different_agent` test).
</objective>
