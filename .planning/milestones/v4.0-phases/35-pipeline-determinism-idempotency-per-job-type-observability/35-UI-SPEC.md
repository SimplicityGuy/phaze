# Phase 35 — UI Design Contract

> Visual and interaction contract for the Pipeline DAG canvas (sketch 001 Variant B "Graph canvas"). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope note:** This phase REPLACES the Phase-34 pipeline UI (`stage_cards.html` + `processing_card.html`) with a single node-edge DAG on an SVG canvas (CONTEXT D-01, LOCKED). The visual layout, dependency model, node anatomy, and live-state examples are already chosen. This document FORMALIZES them into a buildable contract; it does not re-open layout choices.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (server-rendered Jinja2 + HTMX 2.0.7 + Alpine.js 3.15.9 + Tailwind 4.3.0 self-hosted browser build; no build step) |
| Preset | not applicable |
| Component library | none — hand-authored Jinja partials; reuse existing pipeline partial patterns |
| Icon library | inline SVG (Heroicons-style, matching `base.html` nav/theme icons) — no icon font |
| Font | Inter (body/UI, weights 400 + 600); Jura reserved for brand display only (`.font-jura`, not used inside the DAG) |
| Dark mode | `.dark` class on `<html>`, custom variant `@custom-variant dark (&:where(.dark, .dark *))`; every DAG node MUST carry both light and `dark:` classes |
| State store | extend existing `$store.pipeline` Alpine store (`base.html`); NO new store, NO new poll loop, NO SSE |
| Live refresh | existing 5s `/pipeline/stats` HTMX poll + OOB-swap (Phase 34); the DAG re-uses this exact mechanism |

**Stack constraints (LOCKED, CLAUDE.md + CONTEXT):** no React/Vue, no client build pipeline, no new JS framework, no new endpoint beyond what exists. The SVG DAG is rendered once server-side and kept live via store-bound bindings refreshed by the existing poll.

## Spacing Scale

Declared values (all multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Pill padding, icon-to-text gap inside a node |
| sm | 8px | Progress-bar vertical margin, count-to-bar gap, intra-node row gap |
| md | 12px | Node interior padding (`p-3`), node-content column gap (`gap-3`) |
| lg | 16px | Canvas interior padding (`p-4`), gap between page sections |
| xl | 24px | Page section vertical rhythm (`space-y-6` shell) |
| 2xl | 32px | Major layout break above/below the canvas |

Exceptions: **The sketch's `my-1.5` (6px) progress-bar margin is corrected to `my-2` (8px)** to keep every value on the 4px grid. Node corner radius = `rounded-xl` (12px). The 3px stage top-stripe is a hairline accent, not a spacing token.

## DAG Node ("Stage Chip") Anatomy

A node is a `div.node` (rounded-xl, 12px padding, 1px secondary border, 3px stage top-stripe). Min width **130px**, max width **192px**; tier-1 nodes are **176px** (`w-44`). Two interior layouts are permitted:

**Full layout** (Discovery, Analyze, Proposals, Execute — vertical stack):
```
┌─ [3px stage top-stripe] ────────────────┐
│  Stage name (12px/600, stage color)  ·  state pill │   ← row 1: label + pill, space-between
│  27  / 11,428        (18px count, 14px denom)      │   ← row 2: count + denominator (tabular-nums)
│  ▓▓▓░░░░░░░░░░  (5px progress bar, stage fill)     │   ← row 3: per-stage progress bar, my-2
│  [  Run Analysis  ]   (full-width trigger button)  │   ← row 4: trigger (real <button>/<a>)
└────────────────────────────────────────┘
```

**Compact layout** (Extract Metadata, Fingerprint, Scan/Search — horizontal, count left / icon-button right):
```
┌─ [3px stage top-stripe] ──────────────────────┐
│  Stage name          ⟳   │   ← label + count on left, square icon-button right
│  11,428  · DONE pill      │
└──────────────────────────┘
```

Anatomy elements:

| Element | Spec |
|---------|------|
| Top-stripe | 3px bar, stage categorical color, full node width, top edge |
| Stage name | 12px/600, stage color text, never truncates below full word |
| State pill | 12px/600 uppercase, semantic bg-tint (see States); always present, carries the text label of the state |
| Count | 18px/600 tabular-nums, the `done` value |
| Denominator | 14px/400, muted (`text-gray-500 dark:text-gray-400`), rendered as `/ {total}` — **except the tracklist head node, see Denominator Rule** |
| Progress bar | 5px tall, `rounded-full`, track = `bg-gray-200 dark:bg-phaze-panel`, fill = stage color, width = `done/total %`. **Omitted/indeterminate on the no-denominator node** |
| Trigger | a real `<button>` (enqueue) or `<a>` (navigate); full-width in full layout, square icon in compact; never an SVG-only click target |
| Border / elevation | 1px secondary border, no drop shadow at rest; `active` state adds a 1px stage-color ring + soft stage-color glow (`shadow`) |

## The 9 Nodes + Topology

Node/edge topology is authoritative from `35-STAGE-DEPENDENCIES.md`. **Edges are drawn from node anchor points** computed from a single layout constant map — NOT hand-placed per-path `d` strings as in the throwaway sketch.

### Node inventory

| # | Node | Stage color | Layout | Has trigger? | Trigger action |
|---|------|-------------|--------|--------------|----------------|
| 1 | Discovery | cyan | full | yes | `Rescan` → existing scan trigger (POST scan) |
| 2 | Extract Metadata | sky | compact | yes | `Extract Metadata` → `POST /pipeline/extract-metadata` |
| 3 | Analyze | amber | full | yes | `Run Analysis` → `POST /pipeline/analyze` |
| 4 | Fingerprint | teal | compact | yes | `Fingerprint` → `POST /pipeline/fingerprint` |
| 5 | Scan / Search (tracklist head) | rose | compact | yes* | `Run` → existing tracklist scan/search endpoint |
| 6 | Scrape Tracklist | rose | compact | **no** (display-only) | — (per-tracklist trigger lives on `/tracklists`) |
| 7 | Match Discogs | rose | compact | **no** (display-only) | — (per-tracklist trigger lives on `/tracklists`) |
| 8 | Proposals | purple | full | yes | `Generate Proposals` → `POST /pipeline/proposals` |
| 9 | Approve → Execute | green | full | yes (navigational) | `Review queue` → `<a href="/proposals/">` |

\* Scan/Search shows a trigger ONLY if a bulk pipeline-level endpoint already exists. If only per-file endpoints exist, render it display-only like nodes 6–7 (deferred-ideas: "only surface triggers for stages that already have endpoints; net-new endpoints are out of scope"). Planner confirms which.

### Edges (from → to), drawn from anchors

```
Discovery → Extract Metadata
Discovery → Analyze
Discovery → Fingerprint
Discovery → Scan/Search          (tracklist branch head)
Extract Metadata → Proposals     ┐ JOIN — Proposals depends on
Analyze          → Proposals     ┘ Analyze + Metadata ONLY
Proposals → Approve → Execute
Scan/Search → Scrape Tracklist   ┐ sequential tracklist
Scrape Tracklist → Match Discogs ┘ sub-chain (side-branch)
```

**Edge honesty (must be visually verifiable):**
- Fingerprint has **NO** outgoing edge into Proposals. The tracklist subgraph has **NO** edge into Proposals. Only Analyze and Metadata converge into Proposals. A reviewer must be able to confirm this by eye.
- The three file-group siblings (Metadata ∥ Fingerprint ∥ Analyze) and the tracklist head all fan out from Discovery as **parallel** branches — the layout must not let them read as a sequential chain.

### Edge construction contract (anchors, not magic coords)

Define ONE source-of-truth layout map (Jinja context dict or inline JS const), e.g. `NODE_LAYOUT = { "discovery": {x,y,w,h}, ... }`, and the edge list above. A small helper computes each cubic-bézier `d` from `source.rightCenter → target.leftCenter`:

```
M {sx},{sy}  C {sx+Δ},{sy}  {tx-Δ},{ty}  {tx},{ty}
```
where `sx = x+w, sy = y+h/2` (source right-center), `tx = x, ty = y+h/2` (target left-center), `Δ ≈ 60`. Topology is static, so positions live in the layout map (the single place to tune); edge `d` strings are **derived**, never hand-typed. Suggested column layout: col 0 Discovery · col 1 {Metadata, Analyze, Fingerprint, Scan/Search stacked} · col 2 {Proposals, Scrape} · col 3 {Execute, Match}.

## Node States + Visual Treatment

Four states. **Color is never the only signal** — each state also carries a text pill, and disabled nodes carry a reason string + a lock/clock icon.

| State | When | Node treatment | Pill (text + tint) | Count / progress | Trigger |
|-------|------|----------------|--------------------|------------------|---------|
| **complete** | `done == total` (and total > 0) | normal border, full stage stripe | `DONE` — green tint (`bg-green-500/15 text-green-700 dark:text-green-300`) | count shown, bar 100% | enabled (re-run allowed) or icon-only |
| **active** | this stage has in-flight jobs (`agent_active`/`controller_active` > 0 for its function) | 1px stage-color ring + soft stage glow | `{N} ACTIVE` — amber tint (`bg-amber-500/15 text-amber-700 dark:text-amber-300`) | `done / total`, bar at `done/total %`; optional pulse dot | label reads `Running…`, disabled while active |
| **idle** | enabled, no in-flight jobs, `done < total` | normal border | `READY` — sky tint (`bg-sky-500/15 text-sky-700 dark:text-sky-300`) | `done / total`, partial bar | enabled |
| **disabled** | upstream dep unmet OR agent/controller busy | `opacity-60`, muted stripe | `WAITING` / `GATED` — gray tint (`bg-gray-500/15 text-gray-500`) | `done / total` still shown (never blanked) | `disabled` + reason text + lock/clock icon |

## Trigger Gating Contract + Disabled-Reason Copy

Triggers disable on (a) an unmet upstream dependency, or (b) agent/controller busy. Bindings read the extended `$store.pipeline` (no new poll). All trigger buttons keep the existing per-button `x-data="{ loading: false }"` + `@click="loading = true"` + `hx-*` enqueue pattern from `stage_cards.html`.

### Gating predicates per node

| Node | `:disabled` predicate (Alpine) | Reason resolution (first match wins) |
|------|-------------------------------|--------------------------------------|
| Discovery (Rescan) | `loading` | — (root, always actionable) |
| Extract Metadata | `loading \|\| store.discovered === 0 \|\| store.agentBusy > 0` | `No files discovered` → `Agent busy` |
| Analyze | `loading \|\| store.discovered === 0 \|\| store.agentBusy > 0` | `No files discovered` → `Agent busy` |
| Fingerprint | `loading \|\| store.discovered === 0 \|\| store.agentBusy > 0` | `No files discovered` → `Agent busy` |
| Scan / Search | `loading \|\| store.discovered === 0 \|\| store.agentBusy > 0` | `No files discovered` → `Agent busy` |
| Proposals | `loading \|\| store.analyzed === 0 \|\| store.controllerBusy > 0` | `Waiting on Analyze` → `Controller busy` |
| Approve → Execute | `store.approved === 0` (nav link) | `Needs proposals` |
| Scrape · Match | always non-interactive (display-only) | static sub-label `seq · needs tracklist` |

> **Topology correction (carry to planner):** Fingerprint's only hard upstream is **Discovery** (it reads the file on disk, NOT metadata — `35-STAGE-DEPENDENCIES.md`). The Phase-34 `stage_cards.html` gated Fingerprint on `metadataExtracted`; the DAG MUST gate it on `discovered` to match the true (and the visually-drawn) topology. Update the store key the Fingerprint node reads accordingly.

### Exact disabled-reason strings (LOCKED copy)

```
No files discovered      ← discovered === 0 on any file-group / tracklist-head trigger
Agent busy               ← agentBusy > 0  (Metadata, Fingerprint, Analyze, Scan/Search)
Controller busy          ← controllerBusy > 0  (Proposals)
Waiting on Analyze       ← analyzed === 0  (Proposals; the analyze+metadata join not yet satisfied)
Needs proposals          ← approved === 0  (Approve → Execute)
Needs tracklist          ← static sub-label on Scrape / Match (no upstream tracklist yet)
```

### How the reason is surfaced

- **Visible:** when disabled, the button's own label IS the reason (e.g. button text = `Waiting on Analyze`), matching the sketch. For compact icon-button nodes, the reason appears as a node sub-label instead.
- **Screen-reader:** the button carries an `:aria-label` that always names the action and, when disabled, appends the reason in parentheses — e.g. `:aria-label="disabled ? 'Generate proposals (waiting on Analyze)' : 'Generate proposals'"`. The real `disabled` attribute is kept (correct semantics + matches Phase-34 `:disabled`); the informative `aria-label` remains exposed in the accessibility tree. Do not rely on color or the pill alone.

## Counter / Denominator Display Rule (per node)

`done` self-heals from DB-truth on every read (CONTEXT D-03); the maintained Redis counters are a fast cache, DB is truth. Per RESEARCH Q5, the reconcile source is the stage **output table**, NOT the linear `FileRecord.state` — most nodes show `done / total`; **one node has no clean denominator**.

| Node | `done` (DB-truth) | `total` (denominator) | Display |
|------|-------------------|------------------------|---------|
| Discovery | `COUNT(files)` | itself (root) | `{done}`, bar = 100% |
| Extract Metadata | distinct `file_id` in `metadata` | music/video file count | `{done} / {total}` + bar |
| Fingerprint | distinct `file_id` in `fingerprint_results` (completed) | music/video file count | `{done} / {total}` + bar |
| Analyze | distinct `file_id` in `analysis` | music/video file count | `{done} / {total}` + bar |
| **Scan / Search** | distinct `file_id` in `tracklists` | **NONE — no table defines "should get a tracklist"** | **`{done} / —`**: show the done count with an explicit em-dash denominator; **NO determinate progress bar**. If a maintained `enqueued` counter exists, may render `{done} / {enqueued}` as a soft progress hint, but NEVER a fabricated fraction-of-archive. |
| Scrape Tracklist | distinct `tracklist_id` in `tracklist_versions` | `COUNT(tracklists)` | `{done} / {total}` + bar (display-only) |
| Match Discogs | distinct `tracklist_id` in `discogs_links` | `COUNT(tracklists)` | `{done} / {total}` + bar (display-only) |
| Proposals | distinct `file_id` in `proposals` | convergence set: files with BOTH `metadata` AND `analysis` | `{done} / {total}` + bar |
| Approve → Execute | distinct `file_id` in `execution_log` (completed) / `state==EXECUTED` | approved-proposal count | `{done} / {total}` + bar |

**Contract decision (not cosmetic):** the Scan/Search node MUST render its denominator as a literal `—` (em dash) and suppress the filled progress bar, so the UI never fabricates a false total for the tracklist branch. This is the one node where the maintained counter — not a DB denominator — is the only "total" signal (CONTEXT D-02 "maintained counters"). The progress-bar track may still render (empty/indeterminate styling) for visual parity, but no `done/total %` width is computed.

## Live-Update Behavior

Bind to the **existing** `$store.pipeline` + 5s `/pipeline/stats` OOB-swap (CONTEXT). **No new poll loop, no SSE, no blunt re-render of the canvas or buttons.**

### OOB-swap region contract

Mirror the Phase-34 split exactly (the load-bearing lesson from `stats_bar.html` / `stage_cards.html`):

1. **Static, render-once (NOT swapped):** the SVG `<svg>` edge layer and the node chip frames + the trigger `<button>`/`<a>` elements. The trigger subtrees hold `x-data` loading state + in-flight HTMX responses; a blunt swap would clobber an in-flight click or revert an "Enqueued N" message. The static canvas renders once on the full-page `dashboard.html` load.
2. **Live, store-driven:** each node's count, denominator, bar width, state pill, and every trigger `:disabled` read `$store.pipeline`. The 5s `/pipeline/stats` response carries hidden, OOB-swapped `x-init` seed paragraphs (the established `id`-stable pattern) that re-push fresh per-node counts into the store. Alpine re-initializes the swapped seed node and the bindings update — without ever swapping the buttons or the SVG.

**Store extension:** add per-node sub-keys to `$store.pipeline` (e.g. `metadataDone/metadataTotal`, `fingerprintDone/fingerprintTotal`, `analyzeDone/analyzeTotal/analyzeActive`, `tracklistDone`, `scrapeDone/scrapeTotal`, `matchDone/matchTotal`, `proposalsDone/proposalsTotal`, `approved`, `executedDone/executedTotal`) alongside the existing `discovered/analyzed/metadataExtracted/agentBusy/controllerBusy`. Every key MUST be seeded on the initial full-page render (so no binding reads `undefined` before the first poll tick) and refreshed by the OOB seeds — identical to the Phase-34 seed contract.

### Empty / zero / loading states

| Condition | Behavior |
|-----------|----------|
| Fresh DB (`discovered === 0`) | Full topology still renders (it is structural). All nodes show `0`, empty bars, all triggers disabled with `No files discovered` — **except Discovery's Rescan, which stays enabled.** Surface an empty-state hint: heading **"No files discovered"**, body **"Trigger a scan to populate the pipeline."** |
| Pre-first-poll (page load) | Counts/bars/`:disabled` are correct immediately because the store is seeded from the server full-page context — no flash, no jitter. |
| Tracklist branch with no tracklists | Scan/Search shows `0 / —`; Scrape & Match show `0 / {total}` with `seq · needs tracklist`. |
| Enqueue failure | Reuse the existing per-button HTMX response slot; show inline **"Couldn't enqueue. Retry."** in the button's response div (matches existing `*-response` targets). |

## Dark Mode, Responsive, Accessibility

### Dark mode
Every node, edge, pill, bar, and button declares both light and `dark:` classes. Surfaces use the phaze tokens (`bg-white`/`dark:bg-phaze-panel`, `border-gray-200`/`dark:border-phaze-border`, page `bg-white`/`dark:bg-phaze-bg`). Stage colors use the light/dark token pairs in the palette table. The sketch is dark-only and is the dark reference; the light treatment is specified here and is mandatory (the app defaults to `auto` theme).

### Responsive (the SVG DAG cannot reflow)

| Breakpoint | Behavior |
|------------|----------|
| ≥ `lg` (1024px) | Full SVG canvas at its natural size inside the `max-w-7xl` shell. |
| `sm`–`lg` | Wrap the canvas in `overflow-x-auto` with a fixed `min-width` equal to the viewBox width (~1180px). Operator scrolls horizontally; nodes never shrink below 130px. Show a one-line scroll hint. |
| < `sm` (640px) | Hide the SVG canvas; show the **stacked linear fallback** (`<ol>`, below) — the DAG can't usefully shrink onto a phone, so a topologically-ordered vertical list of the same stage chips (no edges) is shown instead. |

### Stacked fallback `<ol>` = also the text-equivalent (one structure, two jobs)
Author a single `<ol>` listing all 9 stages in topological order (Discovery → Metadata → Analyze → Fingerprint → Scan/Search → Scrape → Match → Proposals → Execute), each `<li>` stating name, `done / total` (or `done / —`), current state, and disabled reason. It is `sr-only` (visually hidden) at ≥ `sm` and visible at < `sm`. This gives screen-reader users a full text equivalent of the graph on every viewport AND gives phone users a usable fallback — no duplicate authoring.

### Keyboard & focus order
- All triggers are real `<button>`/`<a>` elements (never SVG `onclick`), so they are natively focusable and Tab-reachable.
- DOM order of the trigger elements MUST equal topological order, so Tab traverses the pipeline Discovery → … → Execute.
- Focus ring: `focus-visible:ring-2 focus-visible:ring-blue-500 focus-visible:ring-offset-2` (brand accent), visible in both modes.
- Node chips themselves are not focusable; only their interactive controls are.

### Color-independent state
Every state is encoded by **text + icon + color**, never color alone: the state pill text (`DONE`/`{N} ACTIVE`/`READY`/`WAITING`/`GATED`), the disabled-reason string, and a status icon (check / spinner-pulse / lock / clock).

### Accessibility of the SVG DAG specifically
- Canvas container: `role="group"` `aria-label="Pipeline stage graph"`.
- SVG edge layer: `aria-hidden="true"` (decorative — dependencies are also conveyed in the `<ol>` text equivalent and node labels).
- Each trigger: real `<button>`/`<a>` with an informative `:aria-label` that includes the disabled reason when disabled (see Gating). Disabled enqueue buttons use the real `disabled` attribute; the navigational Execute link uses `aria-disabled` semantics appropriate to an `<a>`.
- **No** `aria-live` on the per-node counts (the 5s poll would make it chatty). The existing `aria-live="polite"` "Processing" status region is retained for aggregate in-flight feedback; per-node counts are perceivable on demand via the text-equivalent list.
