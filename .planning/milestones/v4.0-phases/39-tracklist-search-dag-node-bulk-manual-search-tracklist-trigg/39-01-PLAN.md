<objective>
Make the DAG the manual control surface for name-based tracklist discovery: turn the
display-only "Scan / Search" head into a triggerable **Search** node whose bulk endpoint
enqueues `search_tracklist` over files that do not yet have a tracklist, gated disabled
until Metadata has produced tags and gated "busy" while a search batch is in flight.

Purpose: Today the only `search_tracklist` trigger is the single-file `POST /tracklists/search`
(`manual_search`) on the Tracklists/Proposals pages; the DAG's "Scan / Search" chip is
display-only. This plan adds the bulk DAG trigger so the operator can launch name-search
over the whole eligible archive from the pipeline graph. Manual only ‚ÄĒ NO auto-trigger
(auto-enqueue is reserved for the Phase-42 recovery pass).

Output: a `POST /pipeline/search-tracklists` HTMX endpoint, a degrade-safe
`get_search_busy_count` service, a `searchBusy` store key, and the Search node trigger
button + gate on the canvas ‚ÄĒ all with co-located TDD tests, full suite green, ‚Č•85% coverage.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator browser ‚Üí POST /pipeline/search-tracklists | Untrusted request crosses into a bulk enqueue. The endpoint takes NO operator-supplied parameters (parameterless POST, like the other pipeline triggers); the eligible set is computed server-side. |
| router/service ‚Üí saq_jobs + tracklists/files tables | New reads: the eligible-set query (files without a tracklist) and the searchBusy in-flight count over saq_jobs on the hot 5s poll. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-39-01 | Tampering/Injection | get_search_busy_count SQL + eligible-set query | mitigate | Both are static SQL with NO operator input interpolated ‚ÄĒ `get_search_busy_count` uses a fixed `text()` SELECT (only literals: split_part, the status allowlist, the function-name constant); the eligible-set query uses ORM `.in_(MUSIC_VIDEO_TYPES)` + `~exists(...)` bound params. Mirrors the Phase-37/t7k static-SQL discipline. |
| T-39-02 | Denial of Service | bulk search enqueue over many files | accept | Enqueuing search over the whole eligible archive is the intended operation; the background task avoids HTTP timeout and the deterministic key `search_tracklist:<file_id>` dedups in-flight re-runs so a double-click/refresh cannot multiply the backlog. Routed to the controller queue (a real consumer), never default. |
| T-39-03 | Denial of Service | searchBusy read on the 5s /pipeline/stats poll | mitigate | The saq_jobs read runs inside a SAVEPOINT (session.begin_nested()); on a missing-table/DB hiccup it degrades to 0 and never raises, and the SAVEPOINT rollback does not expire the dashboard's loaded ORM objects (no plain session.rollback) ‚ÄĒ the poll keeps serving 200 (mirrors get_stage_busy_counts T-t7k-02). |
| T-39-04 | Elevation of Privilege | controller-queue routing | mitigate | resolve_queue_for_task("search_tracklist", ...) is pinned to CONTROLLER_TASKS; a routing regression that sent it to the default queue is caught by the capture assertion `{("controller","search_tracklist")}` (Phase-30 rule). |
| T-39-SC | Tampering | npm/pip/cargo installs | accept | No new packages are installed in this phase ‚ÄĒ endpoint, service, and template changes only use existing deps. No legitimacy gate required. |
</threat_model>
