<objective>
Decouple the agent liveness heartbeat from the SAQ job-dispatch concurrency pool so a worker saturated with multi-hour `process_file` jobs still reports liveness and is never wrongly marked DEAD.

Root cause (CONTEXT.md): `heartbeat_tick` is a SAQ `CronJob` registered in the agent worker, competing for the same `worker_max_jobs=8` dispatch slots as 2-3.6h analysis jobs. All 8 slots stay occupied for hours, so the heartbeat only fires when a long job frees a slot — observed gaps of 50+ minutes against a 300s DEAD threshold. The worker event loop is NOT blocked (Phase 43 moved essentia into a pebble ProcessPool that `process_file` merely `await`s), so a plain asyncio background task on the loop ticks reliably regardless of dispatch saturation.

Purpose: Guarantee liveness independent of job cost. This is distinct from Phase 43 (which bounds how long a job runs) — even a perfectly bounded job would have starved the heartbeat under the cron design.

Output: heartbeat runs as an asyncio background task in `startup(ctx)` (cancelled in `shutdown`); the `CronJob` is removed; full defensive behavior preserved; tests prove starvation-independence + every defensive branch; docs updated.

OUT OF SCOPE (do not touch): the analysis-throughput backlog itself (months of runtime at 8x~3h) — that is Phase 43 + the cloud-burst backlog item. Do NOT modify `process_file`, `pool.py`, or pool/dispatch concurrency. This phase ONLY guarantees liveness regardless of job cost.
</objective>
