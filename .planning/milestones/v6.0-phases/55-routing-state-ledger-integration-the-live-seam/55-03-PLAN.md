<objective>
Wire K8s in as the ONE new branch at the two coordinated fork points keyed on `cloud_target` (D-01),
inside the existing v5.0 `stage_cloud_window` ≤N in-flight window. This is THE live-seam edit — the
phase's highest-risk plan. It depends on Plan 01 (`cloud_target` exists) and Plan 02 (submit seeds
`cloud_phase`). It must NOT break the working a1 rsync flow: when `cloud_target="a1"` the path behaves
byte-for-byte as today.

Two fork points, three load-bearing landmines:
- **Fork A (stage-side, D-01a):** in `stage_cloud_window`, the a1 branch enqueues `push_file`; the new
  k8s branch flips `AWAITING_CLOUD→PUSHING` and stages to S3 instead — reusing the same advisory lock,
  FIFO `FOR UPDATE SKIP LOCKED` claim, and `window/slots` math.
- **Fork B (post-staging, D-01b):** the k8s upload-complete callback `report_uploaded` (which today
  touches NO FileRecord and enqueues NOTHING) must flip `PUSHING→PUSHED` and enqueue `submit_cloud_job`.
- **L1:** `cloud_staging.stage_file_to_s3` commits internally (`cloud_staging.py:119`); calling it
  per-candidate inside the advisory-locked loop would release `pg_advisory_xact_lock` mid-loop and
  re-open the over-stage class. Extract a no-commit core; the cron commits once after the loop.
- **L2:** GATE-1 (online compute agent) is wrong for k8s (no persistent compute agent) — skip it on
  the k8s branch or every k8s file wedges in AWAITING_CLOUD forever. GATE-2 (fileserver) stays.

Purpose: a long k8s file traverses AWAITING_CLOUD→PUSHING→PUSHED→ANALYZED via S3 + Kueue, with the
≤N window draining identically to a1.
Output: the no-commit core, the k8s stage branch, the extended `report_uploaded`.
</objective>
