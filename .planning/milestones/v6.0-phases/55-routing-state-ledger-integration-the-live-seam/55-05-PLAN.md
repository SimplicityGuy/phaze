<objective>
Ship the KROUTE-06 pipeline-dashboard admission-state cards (D-04) — a thin, server-rendered read over
the `cloud_job.cloud_phase` column added in Plan 02. Per 55-UI-SPEC this is a reuse/extension of the
Phase 54 `inadmissible_card.html` pattern, not a net-new visual language. Wave 3: depends on Plan 02
(the `cloud_phase` column + `CloudPhase` enum) and Plan 04 (shares `routers/pipeline.py` +
`services/pipeline.py` — must follow it to avoid a same-wave file conflict).

The cards display per-phase counts (queued_behind_quota / admitted / running / finished) kept live by
the existing 5s `/pipeline/stats` OOB poll. Read-only observability — no CTA, no destructive action.
The locked gating contract: an always-emitted empty `<section>` carrier (stable OOB target); heading +
grid render only when any count > 0; per-tile gated on its own count; degrade-safe `_safe_count` reads
(DB error → 0 → quiet empty carrier). For a1/local deploys cloud_phase is NULL so the card stays quiet.

Two 55-UI-SPEC divergences from naive mirroring, both deliberate: `finished` uses GREEN (not a second
gray — two gray tiles would be indistinguishable; green is the dashboard's done/approved hue), and
`role="alert"` is OMITTED (mirror the inadmissible card's STRUCTURE — section carrier + aria-labelledby
+ OOB — not its alert role; amber + alert stay exclusive to the genuine fault).

Purpose: the operator sees k8s admission progression at a glance.
Output: `get_cloud_phase_counts`, the new card partial, the dashboard mount + stats_bar OOB re-push, dual-context router seeding.
</objective>
