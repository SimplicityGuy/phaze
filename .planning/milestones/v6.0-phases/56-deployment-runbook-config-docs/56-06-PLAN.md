<objective>
Reverse KJOB-05 ("internal CA baked into the Job image") and implement the previously-deferred
KDEPLOY-06 ("ConfigMap/Secret-mounted CA so rotation skips a Job-image rebuild"). The Job image
stops baking a CA entirely; the suspended Job manifest mounts an operator-created K8s Secret
(holding `phaze-ca.crt`) read-only and points `PHAZE_AGENT_CA_FILE` at it.

Why now: the `build-job-runner` job in `.github/workflows/docker-publish.yml` hard-fails on every
`main` push at the "Materialize the internal CA cert" step because the `PHAZE_INTERNAL_CA_CERT`
repo secret is unset — and there is no canonical CA to bake. The internal CA is generated
**per-deployment at runtime** by `cert_bootstrap` and is unique per operator; baking the
maintainer's private CA into a public GHCR image is wrong. Runtime mounting aligns the Job with
how the v5.0 cloud-agent compose already consumes the CA (`${CA_PATH}:/certs:ro`).

Output: a CA-free Job image, a CA-mounting Job manifest, a `kube_ca_secret_name` knob, the
operator CA-Secret runbook in `docs/k8s-burst.md`, knob + revert docs, and the KJOB-05→KDEPLOY-06
decision flip recorded in REQUIREMENTS.md.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator → kube Secret | The CA cert rides an operator-created K8s Secret; phaze references it by name only and never authors or logs it |
| control plane → Job manifest | phaze writes the volume/mount/env referencing the Secret name; it never embeds CA bytes |
| mounted Secret → one-shot pod | The pod reads /certs/phaze-ca.crt read-only; construct_agent_client's st_size==0 guard still fails loud on an empty CA |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-56-06-CA-EMPTY | Spoofing/TLS | mounted CA | mitigate | construct_agent_client's st_size==0 guard (unchanged) fails the one-shot loud on an empty/missing CA, never verify=False |
| T-56-06-CA-LEAK | Information Disclosure | public GHCR image | mitigate | the image no longer bakes any CA, so a per-operator private CA can never leak via a published layer |
| T-56-06-SECRET-NAME | Tampering | kube_ca_secret_name | accept | a plain object name (not a credential); referenced by name, mounted readOnly; no operator free-text reaches the CA bytes |
| T-56-06-SC | Tampering | npm/pip/cargo installs | accept | zero new external packages; manifest is a plain dict |
</threat_model>
